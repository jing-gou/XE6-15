# MVP功能边界

## 产品定位

一个基于 **GitHub 生态**上的时间管理 App，MVP阶段面向 XEngineer 训练营学生，帮助解决"不知道时间花在哪、该先做什么、进度如何"的核心问题。

## 目标用户

XEngineer 训练营学生个人。基本特征见 [用户画像及用户故事](7.9-用户画像及用户故事.md)。

### 核心痛点

| 痛点 | 描述 |
|------|------|
| 进度不可见 | 投入、推进实际记录缺失，和项目推进没有关联，并不清楚项目当前实际推进状态 |
| 拆解能力弱 | 任务拆解与规划能力偏低，无法合理设定任务产出预期与任务投入优先级决定 |
| 时间管理无保证 | 缺乏项目、任务的时间管理视角，对于投入产出进度无保证 |

## MVP 闭环：计划 → 追踪 → 感知 → 汇报

时间管理的核心闭环只需 4 个用户场景即可串起：

```
计划层          执行追踪层        感知&调整层        汇报层
─────────────────────────────────────────────────────────
场景8 任务拆解   场景1 时间预估    场景2 偏离预警     场景4 日报生成
                场景1 耗时追踪    场景2 优先级重排
```

一条完整的 MVP 用户旅程：

> 早上，大任务被自动拆成可执行的子步骤 → 每个子步骤有预估工时 → 开始执行后系统持续追踪实际耗时 → 偏离时预警并建议重排优先级 → 晚上自动生成日报，列出完成/进行中/阻塞/明日计划。

## 做什么（MVP 必做）

### 场景1、2：时间预估 → 偏离预警 → 优先级重排

**闭环环节**：执行追踪 + 感知 & 调整

**做什么**：
- 支持用户为 Issue 打上预估耗时 Label（如 `est:4h`）
- 持续追踪 Issue 在 Projects 各列中的停留时长（实际耗时）
- 当实际耗时明显超过预估 Label 所标工时，通过 Issue Comment 自动留言预警并 @mention 用户
- 当某个 Issue 发出进度偏离预警后，系统根据所有 Issue 的 `priority` Label（`P0`/`P1`/`P2`）、预估工时、实际已投入时间、Milestone 截止日期，自动给出今日优先处理建议

**兼容性要求**：Projects 看板在营规范中为可选，需支持无 Projects 模式——基于 Milestone 下的 Issue 列表做优先级重排。

### 场景8：大 Issue 拆解为 Task List

**闭环环节**：计划

**做什么**：
- 当用户将一个标题或 Body 较为宏观的 Issue 拖入 "In Progress" 列时，（系统分析 Issue 描述内容），手动（自动）生成细粒度执行步骤清单
- 以 GitHub Task List（`- [ ]` Checkbox）形式追加到 Issue Body 中
- 每完成一个子步骤（勾选 Checkbox 或关闭子 Issue），父 Issue 进度可见推进
- 所有子步骤 Checkbox 勾选完毕后，父 Issue 自动关闭

### 场景4：基于 GitHub 活动自动生成日报

**闭环环节**：汇报

**做什么**：
- 每日固定时间扫描用户当天在 GitHub 上的活动记录：已关闭的 Issue、已合并/已提交的 PR、Issue Comments 和 PR Review Comments 中的讨论要点、Projects 看板卡片列变更
- 自动生成结构化日报草稿，按"今日完成"、"进行中"、"遇到阻塞"、"明日计划"四个板块组织
- 每条内容附带对应的 GitHub 链接
- 支持一键发起 `daily-report` 类型的 Discussion 帖

**营规范依据**：每人每天必须在所在导师群发送日报，格式为「人物 + 事件」，有 PR / Issue 等产出必须附解释与链接。缺日报视为危险信号——日报不是"nice to have"，是刚性需求。

## 可选包含

### 场景3：卡点记录 → 结构化求助模版 → @Mentor

**闭环环节**：感知 & 调整（卡点是时间管理的关键信号）

**做什么**：
- Issue 中提供"🆘 求助卡点记录"入口，自动生成结构化 Comment 模版
- 引导填写：问题现象（错误日志）、已尝试方案（Checklist）、相关 PR/Commit 链接、卡住时长
- 填完后自动打上 `help-wanted` 和 `blocked` Label，并 @mention 指定导师或队友
- 卡点解决后，过程记录可沉淀为 Issue 知识资产

**营规范依据**：每日站会强制要求汇报"遇到什么卡点、需要什么支持"。

## 明确不做（MVP 范围外）

| 场景 | 不做的理由 |
|------|-----------|
| 场景5：会议全流程推进（会前议程/会中控时/会后纪要） | 是协作工具场景而非时间管理核心；营已有独立会议规范，系统化并非第一优先级；且场景复杂度高，不适合 MVP 阶段 |
| 场景6：PR Review 反馈按 Label 分级 → 自动创建返工 Issue | 偏代码质量/协作流程，离"时间管理"核心较远；PR diff 分析技术复杂度高 |
| 场景7：会议纪要/Discussion → 自动创建 Issue 并排程 | 任务"录入"环节可被场景8（任务拆解）部分覆盖；MVP 阶段录入先保持手动 |

---

## 验收标准

MVP 通过的标准：

- [ ] 用户可为 Issue 打预估耗时 Label，系统能追踪 Issue 在各列的停留时长
- [ ] 当实际耗时超过预估时，系统自动在 Issue 下留言预警并 @mention 用户
- [ ] 系统能基于优先级、预估工时、Milestone 截止日期给出今日优先处理建议
- [ ] 用户将宏观 Issue 拖入 "In Progress" 后，系统自动生成 Task List 拆解
- [ ] Task List 子步骤勾选完毕后，父 Issue 自动关闭
- [ ] 每日固定时间，系统自动扫描用户当天 GitHub 活动并生成结构化日报（今日完成/进行中/阻塞/明日计划）
- [ ] 日报支持一键发起 Discussion 帖
- [ ] 核心路径在无 Projects 看板的环境下也能走通
