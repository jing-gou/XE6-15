## 一、产品定位与 MVP 范围

本工具面向 **XEngineer 训练营成员**，解决一个问题：

> GitHub 记录了项目对象和提交过程，却没有直接回答"我今天先推进什么、哪个 PR 要处理、计划被打断后怎么重新排"。

MVP 跑通一条最短闭环：

```
GitHub 登录 → 选择仓库 → 导入个人任务池 → 查看今日优先任务（含排序理由）→ 状态变化后自动重排
```

三个 Proposal 的依赖关系：

```
GitHub Issue / PR / Review Request
        │
        ▼
  Proposal 01：任务池导入 ── 把散落在各处的个人行动收进一个池子
        │
        ▼
  Proposal 02：优先级排序与展示 ── 告诉我今天先做什么，并解释为什么
        │
        ▼
  Proposal 03：任务自动重排 ── 计划失效后，自动给出新的可执行计划
        │
        ▼
  用户确认，回到 GitHub 执行
```

**MVP 硬边界**：只读 GitHub、不写回、不做团队视图、不引入 GitHub 以外的数据源。

## 二、核心数据模型

### 2.1 任务对象（从 GitHub 同步）

| 字段 | 类型 | 来源 | 说明 |
|------|------|------|------|
| `id` | string | 系统生成 | 任务唯一标识 |
| `github_type` | `issue` \| `pr` \| `review_request` | GitHub | 区分三种行动类型 |
| `github_number` | number | GitHub | Issue/PR 编号 |
| `github_url` | string | GitHub | 跳转回原页面处理 |
| `title` | string | GitHub | 标题 |
| `repository` | string | GitHub | 所属仓库 full_name |
| `milestone` | string \| null | GitHub | 关联 Milestone 标题 |
| `labels` | string[] | GitHub | 标签列表 |
| `github_status` | `open` \| `closed` \| `merged` | GitHub | GitHub 侧状态 |
| `created_at` | datetime | GitHub | 创建时间 |
| `updated_at` | datetime | GitHub | 最后更新时间 |

### 2.2 个人计划信息（用户侧，不写回 GitHub）

| 字段 | 类型 | 来源 | 说明 |
|------|------|------|------|
| `priority` | `P0` \| `P1` \| `P2` \| `unset` | 用户手动设置 | 个人优先级判断 |
| `hard_deadline` | date \| null | 用户设定 / Milestone 推算 | 硬截止日期 |
| `estimated_hours` | number \| null | 用户手动填写 | 预估投入时间 |
| `manual_order` | number \| null | 系统记录 | 用户拖拽后的手动位置 |
| `aging_days` | number | 系统计算 | 自导入或上次更新以来的天数（用于无 DDL 老化上浮） |

### 2.3 任务运行态（系统驱动）

| 字段 | 类型 | 说明 |
|------|------|------|
| `work_status` | `on_progress` \| `pending_review` \| `done` | **主状态** — 决定是否参与可执行计划 |
| `status_reason` | `changes_requested` \| `blocked` \| `missing_estimate` \| `merged` \| `closed_unmerged` \| null | **状态原因标签** — 解释为什么处于当前状态/位置 |
| `plan_bucket` | `today` \| `planning_pool` \| `pending_review` \| `blocked` \| `unestimated` \| `done` | **计划位置** — 任务在界面中的归属区域 |

### 2.4 每日容量

| 字段 | 类型 | 说明 |
|------|------|------|
| `date` | date | 日期 |
| `total_hours` | number | 用户设定的当日可用总时长 |
| `used_hours` | number | 已完成任务的预计投入之和（不要求用户补录实际工时） |
| `remaining_hours` | number | `total_hours - used_hours` |

## 三、主状态 & 计划位置 —— 核心概念

### 3.1 三个主状态（work_status）

用户只需判断"我现在要不要行动"：

```
On Progress ──→ 需要我行动，参与排序和重排
Pending Review ──→ 我已提交/修改完，等别人 Review，离开可执行计划
Done ──→ 当前没有我需要做的事，退出任务池
```

状态流转规则：

| 从 | 到 | 触发条件 |
|----|----|----------|
| On Progress | Pending Review | 用户提交 PR / 完成返工并请求 Review |
| Pending Review | On Progress | 收到 `changes requested` 或明确的未解决修改意见 |
| On Progress | Done | Issue 关闭 / PR 合并 / PR 关闭未合并 |
| Done | On Progress | Issue 或未合并 PR 被 Reopen（保留原有优先级、DDL、预估、手动顺序） |
| On Progress | Blocked（位置） | 用户标记阻塞 / GitHub 出现 `blocked` label |
| Blocked | On Progress | 用户标记解除阻塞 |

**关键规则**：
- `Approved` 或普通评论 **不改变** Pending Review 状态
- PR 合并 → Done + "已合并"；PR 关闭未合并 → Done + "已关闭未合并"
- `blocked`、`changes_requested` 等不新增主状态，只作为 `status_reason` 标签

### 3.2 六个计划位置（plan_bucket）

| 位置 | 含义 | 是否参与排序 |
|------|------|:--:|
| `today` | 今日计划，当前可执行 | ✅ |
| `planning_pool` | 规划池，后续待安排 | ✅（在池内） |
| `pending_review` | 待评审区域，等别人 | ❌ |
| `blocked` | 阻塞区域，暂时无法推进 | ❌ |
| `unestimated` | 待估算，缺预估用时 | ⚠️ 展示优先级但不进今日计划 |
| `done` | 已结束 | ❌ |

## 六、核心交互流程

### 6.1 排序算法（Proposal 02）

用户点击排序（或触发重排）后，系统逐层执行：

```
第一步：紧急置顶
  └─ DDL ≤ 24h 的任务直接提到最前面
  └─ 紧急任务之间按 DDL 近 → 远

第二步：按优先级分组
  └─ P0 → P1 → P2
  └─ 未设置优先级 → 归入 P2 组末尾

第三步：组内按 DDL 排
  └─ 有 DDL：截止日期近 → 远
  └─ 无 DDL：按老化上浮
       ·  < 7 天 → 不动
       ·  7-14 天 → 轻微上浮
       ·  14-21 天 → 明显上浮
       ·  > 21 天 → 大幅上浮

第四步：同 DDL（或同老化档）内按预估用时排
  └─ 用时短 → 长
  └─ 未填预估 → 排该档最后
```

**关键约束**：
- 同条件下顺序稳定——重复排序不随机跳变
- Issue、PR、Review Request 混排，不分组
- 拖拽过的手动顺序在同一 DDL 分组内保持
- 更早 DDL 可覆盖手动顺序，但需展示变化原因

### 6.2 自动重排流程（Proposal 03）

```
状态变化发生（GitHub Webhook / 用户操作）
        │
        ▼
  判断 work_status 变化
        │
   ┌────┼────┬──────────┐
   ▼    ▼    ▼          ▼
 提交  收到  任务    截止/阻塞
 PR   changes 完成    变化
   │    │     │        │
   ▼    ▼     ▼        ▼
Pending 回到  Done   plan_bucket
Review  On     或    更新
       Progress Blocked
   │    │     │        │
   ▼    ▼     ▼        ▼
 离开  参与   退出    重新
 执行  重排   执行    分配
 计划         计划
```

**当日变化 vs 跨日变化**：同一套体验。不管是下午 PR 被退回，还是第二天早上打开 App，用户看到的都是"新计划 + 变化原因"。

### 6.3 用户手动调整规则

```
用户拖拽任务
      │
      ▼
记录 manual_order（相对位置）
      │
      ▼
下次排序/重排时：
  ├─ 同一硬截止分组内 → 保持手动顺序
  └─ 某任务硬截止提前 → 可覆盖手动顺序，展示"因硬截止变为今天而提前"
```

### 6.4 容量扣减规则

```
任务完成（work_status → done）
      │
      ▼
按原预计投入扣减今日 used_hours
      │
      ▼
不要求用户填写实际工时
不根据 Commit 时间或在线时长猜测
```

## 七、变化摘要规范

### 7.1 触发条件

| 变化 | 展示内容 |
|------|----------|
| 新任务进入今日计划 | `+ [任务] 新进入今日计划` + 原因 |
| 任务离开今日计划 | `- [任务] 移回规划池` + 原因 |
| 任务顺序变化 | `↑/↓ [任务] 顺序变化` + 原因 |
| 任务状态变化 | `🔄 [任务] 从 [旧状态] 变为 [新状态]` |
| Pending Review ↔ On Progress | 显示具体触发原因（提交 PR / changes requested） |
| Done → On Progress (Reopen) | `🔄 [任务] 已 Reopen，回到进行中` |

### 7.2 展示规则

- 有变化时在今日计划页底部显示，默认折叠、显示红点和变化数量
- 用户可展开查看详情
- 变化摘要展示最近 24h 内的变化
- 用户关闭或已读后，红点消失

## 十、MVP 验收场景（Demo 走查用）

### 场景 1：导入任务池（Proposal 01）

1. 用户用 GitHub 登录
2. 选择 `1024XEngineer/XE6-15`
3. 系统同步与用户相关的 Issue、PR、Review Request
4. 任务池页展示所有任务，去重、无遗漏
5. 每项保留 GitHub 链接，点击可跳回

### 场景 2：设置优先级并排序（Proposal 02）

1. 用户为 5 项任务设置 P0/P1/P2
2. 为其中 3 项填写预估用时
3. 点击排序，列表按 紧急置顶 → P0 → P1 → P2 → 未设置 排列
4. 同组内有 DDL 的按截止排，无 DDL 的按老化上浮
5. 点开任意任务，看到可逐条核对的排序理由
6. 拖拽一项任务调换位置 → 该顺序被记住
7. 导入新任务 → 新任务自动插入正确位置，拖过的保持不动

### 场景 3：PR 被退回后的自动重排（Proposal 03）

1. 用户提交 PR → 任务从今日计划消失，出现在待评审
2. PR 收到 `changes requested` → 任务自动回到今日计划
3. 今日容量不足 → 一项较低优先任务移回规划池
4. 变化摘要自动展示：+ 什么回来了、− 什么移出了、原因是什么
5. 用户无需逐项确认，直接按新计划执行

### 场景 4：跨日未完成

1. 今天的任务 B 和 C 未完成
2. 第二天打开 App，今日计划自动更新
3. B（DDL 更早）留在今天，C 移入规划池
4. 变化摘要说明原因："B 因硬截止更早进入今日计划"

### 场景 5：硬截止容量冲突

1. 今日剩余 2h，但两项截止今天的任务共需 3h
2. 弹出容量冲突界面，展示缺口
3. 用户选择一个处理方向后，计划继续

### 场景 6：缺少信息

1. 某项任务缺少预估用时
2. 它出现在"待估算"区域，展示优先级但不进今日计划
3. 用户补充预估后，自动参与容量安排

## 十一、MVP 不做清单（明确边界）

| 不做 | 原因 |
|------|------|
| 团队/组织管理视图 | MVP 只验证个人决策层 |
| 写回 GitHub（关闭 Issue、合并 PR 等） | 先证明读+排有价值 |
| 多数据源（日历、邮件、聊天等） | GitHub-only 假设尚未验证 |
| AI 黑盒排序 | 用户需要可核对的透明规则 |
| 自然语言交互（"帮我排优先级"） | 本期只做手动设定 |
| 长期倒排 / 甘特图 / 日历 | 先验证"今天做什么" |
| 日报 / 周报 / 通知推送 | 完成状态只作为重排输入，不另做输出产品 |
| 移动端 | 先用 Web 验证核心闭环 |
| 跨多仓库同时管理 | 先跑通一个仓库 |
| 用户实际工时记录 | MVP 按预计投入扣减容量 |
| 自动拆分 Issue | 当前数据不足 |
| AI 猜测截止时间 | 信息缺失时明确暴露，不让 AI 编造 |
