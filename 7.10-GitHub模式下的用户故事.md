# GitHub模式下的用户故事
### 用户画像
训练营学生个人
### 基本特征
- 正在参加 X Engineer 训练营
- 使用GitHub进行协作，并且依赖Milestone、Projects、Issue、PullRequest等过程管理功能。
- 有一定技术基础，但仍在快速学习新知识
- 需要在短时间内完成一个项目
- 每天面对学习、开发、文档、会议、复盘等多种任务
- 可能需要提交日报、周报或阶段性汇报
- 不懂得会议组织、产品设计、需求分析、架构推导和优先级判断等实际项目知识

### 用户故事

#### 细颗粒度任务时间规划
**用户角色：** 作为一名刚刚进入实习阶段的Xengineer

**用户故事：** 虽然每日组会给我分配了任务，但是落实到细颗粒度的任务（Issue）上我没有截止时间的概念，也就没有了紧张感，导致原本可以快速完成的工作被我拖延到了下班才完成或者下班都完不成。

#### Issue 进度偏离时间预估的预警
**用户角色：** 作为一名参加 XEngineer 训练营、缺乏真实工程项目经验的学生

**用户故事：** 当我在 GitHub 上为某个 Issue 打上了预估耗时的 Label（如 `est:4h`），并将其拖进 Projects 看板的 "In Progress" 列开始开发后，常常因为低估难度而导致实际投入远超预估，Issue 却始终卡在同一列没有推进。系统会持续追踪每个 Issue 的预估耗时 Label、在 Projects 各列中的停留时长，并结合 Milestone 的截止日期计算出实际进度与预期的偏差。当某个 Issue 在 "In Progress" 列的停留时间明显超过预估 Label 所标工时，系统会通过 Issue Comment 自动留言预警、在 Projects 看板上对该卡片标红高亮，并 @mention 我提醒关注。同时，系统会对比该 Issue 所在 Milestone 下其他 Issue 的进度，提示该任务的延期是否会影响整体 Milestone 的按时交付，帮助我在项目中期就能及时调整，而不是等到 Milestone 截止日临近才发现大量 Issue 堆积未完成。

#### 场景2：偏离预警后在 Projects 看板上的优先级重排
**用户角色：** 作为一名面对多个 Issue 同时堆积、难以判断优先级的训练营成员

**用户故事：** 当系统对某个 Issue 发出进度偏离预警后，我希望它不只是告诉我"这个任务延期了"，而是能帮我重新审视整个 Projects 看板上的任务布局。系统会根据所有 Issue 的 `priority` Label（如 `P0`/`P1`/`P2`）、预估工时 Label、实际已投入时间、Milestone 截止日期以及 Issue 之间的依赖关系（通过 `Depends on #xxx` 或 "Blocked by" 关联），自动给出调整建议：哪些 Issue 必须提升优先级并纳入今日重点攻坚，哪些低优先级 Issue 可以暂时移入 "Icebox" 列或在 Projects 中标记为 "Later"，哪些 Issue 的预估工时需要上调并同步更新 Label。最终在 Projects 看板上生成一个清晰的"今日行动项"视图，让我集中精力完成对项目交付影响最大的工作，而不是在多个 Issue 之间无方向地切换。

#### 场景3：卡点记录 → Issue 结构化求助模版 → @Mentor
**用户角色：** 作为一名遇到技术卡点但不擅长清晰描述问题的训练营学生

**用户故事：** 当我在某个 Issue 上卡住很久、尝试多个方案仍未解决时，常常只在脑子里着急，没有在 GitHub 上留下任何记录。等到想向导师求助时，我需要重新翻找代码、翻阅 Discussion 历史和个人聊天记录来回忆卡在哪里，沟通成本很高。系统会在 Issue 中提供一个"🆘 求助卡点记录"按钮，点击后自动生成结构化的 Issue Comment 模版，引导我填写：**问题现象**（附带错误日志代码块）、**已尝试方案**（Checklist 形式逐条列出）、**相关 PR/Commit 链接**（自动引用该 Issue 关联的分支和提交）、**卡住时长**（从 Issue 进入 "In Progress" 列开始计算）。当信息填完后，系统会将该 Issue 自动打上 `help-wanted` 和 `blocked` Label，在 Issue 标题追加 `[求助]` 前缀，并通过 @mention 通知我指定的导师或队友。同时，如果组织中存在相关技术栈的 Discussion 问答帖，系统会自动在 Issue Comment 中引用推荐，方便我从社区已有讨论中寻找线索。卡点解决后，整个过程记录可作为复盘材料沉淀为该 Issue 的知识资产，方便后续遇到类似问题的队友参考。

#### 场景4：基于 GitHub 活动自动生成日报
**用户角色：** 作为一名需要向导师进行日报汇报但不知如何总结的训练营学生

**用户故事：** 我每天在 GitHub 上做了很多事情——提交 Commit、发起 PR、在 Issue 下讨论、移动 Projects 看板卡片、Review 队友代码——但这些活动分散在不同 Repository 的各个角落，到写日报时我一片空白。系统会在每日固定时间扫描我当天在 GitHub 上的所有活动记录：**已关闭的 Issue**（标记完成的任务）、**已合并/已提交的 PR**（代码产出）、**Issue Comments 和 PR Review Comments 中的讨论要点**（遇到的问题与决策）、**Projects 看板上卡片列变更**（任务推进状态）、**新创建的 Issue 和 PR**（待跟进事项）。然后自动生成一份结构化的日报草稿，按"今日完成"、"进行中"、"遇到阻塞"、"明日计划"四个板块组织，每条内容都附带对应的 GitHub 链接供导师点击验证。我可以直接在 GitHub Discussion 或 Issue Comment 中粘贴这份日报，也支持一键发起一个 `daily-report` 类型的 Discussion 帖供导师团队查看。

#### 场景5：基于 GitHub Issues/PRs/Discussions 的会议全流程推进
**用户角色：** 作为一名缺乏会议组织经验、需要主持团队站会或迭代复盘会的训练营学生

**用户故事：** 作为会议组织者，我常常不知道会议该围绕什么展开。系统在**会前**会自动扫描团队 Repository 中的待处理数据：筛选当前 Milestone 下的未关闭 Issue、处于 "In Review" 列的 PR、标有 `discussion-needed` Label 的 Issue、近期活跃的 Discussion 帖，生成一份会议议程草稿，明确每个议题关联的 GitHub 链接、预计讨论时长和需要决策的关键点。

**会议进行中**，系统在 GitHub Projects 的会议专用视图上展示议程看板，每个议题对应一张卡片，附带预计讨论时间倒计时。当讨论在某个 Issue 的评论区发散或超时时，系统会在该 Issue 下自动追加 Comment 提醒"当前讨论已超出预计时间，建议记录为待决议题并进入下一项"。同时，系统会检测讨论是否偏离该 Issue 的原始描述和目标用户场景，及时给出拉回提示。

**会议结束后**，系统根据会议期间 Issue Comments 中的讨论记录和 Projects 看板上的决议操作，自动生成一份会议纪要，发布为一个新的 Discussion 帖或会议总结 Issue。纪要中包含：**已形成的结论**（附相关 Issue/PR 链接）、**待解决问题**（自动创建新 Issue 并打上 `follow-up` Label）、**每个待办的负责人 @mention、预估工时 Label 和关联的 Milestone 截止日期**。新创建的 Issue 会自动加入 Projects 看板的 "To Do" 列，确保会议中的每一条决议都有对应的 GitHub 任务卡片追踪落地。

#### 场景6：PR Review 反馈按 Label 分级 → 返工 Issue 自动创建
**用户角色：** 作为一名参与团队协作开发、提交 PR 后收到大量 Review 意见的训练营学生

**用户故事：** 我的 PR 提交后，Reviewer 在 "Files changed" 页面留下了十几条 Review Comments，有的标记为 `Request changes`（必须修改），有的只是一些建议性评论。我面对这些反馈时容易陷入两种极端：要么每条都改导致返工时间失控，要么只改明显错误忽略了 Reviewer 的真实关切实点。系统会在 PR 收到 Review 后，自动分析所有 Review Comments，按严重程度分级并打上对应的 Review 标签：**🔴 `must-fix`**（阻断性反馈，不修改不能合入）、**🟡 `should-fix`**（逻辑或设计问题，建议修改但可协商）、**🟢 `nice-to-have`**（代码风格或优化建议，可后续处理）。分级完成后，系统自动在 PR 的描述区域生成一份"返工清单"Checklist，并为每个 `must-fix` 和关键 `should-fix` 项自动创建关联的子 Issue，挂在该 PR 的 "Development" 链接下，同时根据反馈条目数量和历史同类修复数据给出每个子 Issue 的预估返工耗时 Label 建议。这些返工 Issue 自动进入 Projects 看板的 "To Do" 列并关联到当前 Sprint 的 Milestone，确保返工工作不再游离于项目管理之外。

#### 场景7：从会议纪要/Discussion 转录自动生成 Issue 并排程
**用户角色：** 作为一名不想手动逐字录入任务、容易遗漏待办事项的"懒惰"实习生

**用户故事：** 每次会议结束后，团队 Discussion 里或项目 Wiki 上会留下一份会议纪要，里面列出了我的名字和若干任务，但我没有动力把它们一条条手动创建成 Issue——结果就是过两天完全忘记自己该做什么。系统支持我上传会议纪要 Markdown 文件或粘贴 Discussion 帖链接，自动解析其中提到我 GitHub 用户名的待办段落，识别每条待办的内容、关键上下文和隐含的 urgency。然后按识别出的优先级，批量创建 GitHub Issue，自动填入结构化内容：**标题**（任务摘要）、**Body**（包含原始会议记录引用链接、任务背景描述和验收标准）、**Assignee**（自动分配给我）、**Label**（根据内容推断打上对应模块标签如 `frontend`/`backend`/`docs` 和优先级标签 `P0`/`P1`/`P2`）、**Milestone**（关联当前活跃的 Milestone）。所有新 Issue 自动加入 Projects 看板的 "To Do" 列，并按优先级自上而下排列——我打开看板的那一刻，今天该做什么就已经一目了然，无需任何手动打字。

---

#### 场景8：大 Issue 自动拆解为 Task List 或子 Issue
**用户角色：** 作为一名不善于将大型任务分解为可执行小步骤、导致进度追踪颗粒度太粗的训练营学生

**用户故事：** 我创建或分配到的 Issue 通常是一个模糊的大目标，比如"实现用户登录模块"，我把它拖到 "In Progress" 后就埋头苦干好几天，期间 Issue 没有任何状态更新，自己也说不清完成了百分之多少，到了 Milestone 检查点时才发现远未完成。系统在我将一个标题或 Body 较为宏观的 Issue 拖入 "In Progress" 列时，会主动分析该 Issue 的描述内容和关联的 Milestone 目标，自动生成一份细粒度的执行步骤清单，以 GitHub Task List（`- [ ]` Checkbox）形式追加到 Issue Body 中，或对特别复杂的任务自动创建多个子 Issue 并通过 GitHub 的 "Sub-issue" 功能链接到父 Issue。

每完成一个子步骤，我在 Issue 中勾选对应的 Checkbox 或关闭一个子 Issue，Projects 看板上父 Issue 的进度条就会自动推进。这样，一个原本无法追踪进度的"黑洞式"大任务变成了可视化的分步推进：每勾选一项就是一次小小的里程碑，让我清晰地看到自己走了多远、还剩几步、是否需要调整后续子任务的执行顺序。当所有子步骤 Checkbox 被勾选完毕，父 Issue 自动关闭并移入 "Done" 列，Milestone 进度条也同步更新——整个过程让我从"大概感觉做了一半"变成"确知已完成 6/8 子任务"。
