## 一、开场钩子（2min）

### 1.1 一句话定义（30s）

> VoiceLife 声活是一款语音优先的日程提醒工具。你随口说、它按时提醒、你随口回应——全程不用动手。

### 1.2 现场互动（1min）

**主持人提问**："现场有多少人，曾经脑子告诉自己要买一样东西，但回到家才发现忘了？"

（举手的应该不少。）

**追问**："好，在座的有没有人把这事发到微信传输助手的？"

（很多人会举手或笑。）

> "这恰恰说明，大家明明在意这些安排，却找不到一个比'自己记住'或'发微信给自己'更趁手的工具。VoiceLife 就是为这个做的。"

### 1.3 一句话预告（30s）

> "接下来 30 分钟，我会带着大家看——我们面临的问题是什么，我们怎么做的产品设计，以及这套设计怎么落到工程里。"

## 二、问题与用户（3min）

### 2.1 三个痛点场景（1.5min）

**场景一：脑子里记不住**

早上发现牙膏没了，告诉自己下班买。下班后按惯常路径上楼，第二天又没牙膏。来回几次，才在某一天路过楼下超市时终于想起——偶然解决，不是工具解决。

**场景二：掏出手机成本高**

通勤路上、做饭时、开会前后，想记一件事——掏手机 → 解锁 → 找 App → 打字。很多时候心想"算了等会儿再记"，结果忘了。

**场景三：提醒来了没法随口回**

手机闹钟/GUI 提醒响了，但此时手是湿的（做饭）、抓着扶手（公交）、或在和别人说话——想"稍后提醒"还得腾出手点屏幕。

### 2.2 共同问题（30s）

> 不是用户缺乏工具，而是**现有的工具在设计上天然假设用户有手、有屏幕、有注意力**。现实生活中大量时刻不具备这些条件。

### 2.3 目标用户画像（1min）

| 用户类型 | 典型场景 | 核心诉求 |
| --- | --- | --- |
| 日常记录者 | 下班顺路买东西、拿快递 | 快速记住、到时提醒 |
| 忙碌的规律生活者 | 吃药、例会、锻炼 | 按时提醒、不遗漏 |
| 多任务并行者 | 开会、通勤、做饭同时要处理安排 | 不用腾手、随口回应即可 |
| 无屏设备用户 | 使用智能音箱、穿戴设备等 | 纯语音交互 |

## 三、竞品速览与机会点（3min）

### 3.1 现有做法一览（1.5min）

| 竞品/做法 | 优势 | 不足 |
| --- | --- | --- |
| 手机闹钟/提醒 | 简单直接 | 只响铃，不知道是啥事；响应需手动操作 |
| 智能语音助手（Siri/小爱） | 可语音创建提醒 | 提醒到了仍是 GUI 通知，响应仍需手动；周期修改复杂 |
| 日历 App（系统日历/Google） | 日程结构完善 | 创建靠手动填写，语音接口薄弱 |
| 企业微信/钉钉日历 | 团队协作好 | 语音+提醒的闭环不完整，面向团队非个人 |
| 番茄 TODO 等工具 | 专注场景好 | 不是日常生活场景工具，交互依赖屏幕 |

### 3.2 机会点（1min）

- **语音助手只在"记"这一步用了语音，提醒和修改仍回到 GUI**。没有产品把创建、提醒、修改三个动作都用语音串起来。
- **大多数提醒工具假设用户在看屏幕**。无屏或不便看屏的场景长期空白。
- 所以我们不是在和一个产品竞争，而是在一个**语音闭环**的空地上做第一个产品。

> 一句话：Siri 会帮你记账单提醒，但提醒响了你得掏出手机点。VoiceLife 不用。

## 四、产品方案：核心设计（5min）

### 4.1 产品链路：三步闭环（1min）

> **说出来 → 到点听 → 随口改**

```
用户说出安排 ──→ 时间到了语音提醒 ──→ 用户随口回应 ──→ 必要时修改
    (记)              (触)               (回)            (变)
```

三个 Proposal 对应链路上的三个结点：

| Proposal | 链路结点 | 一句话 |
| --- | --- | --- |
| P1：语音创建与查询 | 说出来 | 按住说话，松开记好。随时问"我要干什么来着？" |
| P2：提醒触发与即时响应 | 到点听 | 时间到了告诉你什么事，你可以打断、推迟、追问 |
| P3：已有日程变更与周期控制 | 随口改 | 时间变了说一句就能改，改错了15分钟内撤销 |

### 4.2 P1：语音创建与查询（1min）

**做什么**：用户通过语音创建两种事项——

- **日程**（有时间段，如"明天下午三点到五点开产品会"）
- **提醒事项**（仅有时间点，如"晚上八点吃药"）

**怎么查**：随时语音查询——

- "今天有哪些安排？"
- "下班要干什么来着？"

**设计要点**：这不是聊天机器人，是专注记录生活的工具。只记录和提醒，不闲聊。

### 4.3 P2：提醒触发与即时响应（1.5min）

**触发分为两级**：

| 类型 | 时间 | 行为 | 是否可延迟 |
| --- | --- | --- | --- |
| 预告 | 开始前 15min | 短铃声 + 语音简要播报 | 不可延迟 |
| 提醒 | 开始时 | 语音播报 + 持续铃声 | 1min 无回应则自动推迟 |

**用户能做什么**（四条响应路径）：

1. **打断并确认**："小鹏小鹏，我知道了" → 提醒结束
2. **推迟**："十分钟后再提醒我" → 10min 后再次提醒。无截止时间的提醒事项可无限推迟，有截止时间的日程推迟不超过截止时间
3. **追问详情**："会议地点在哪？" → 系统从日程数据中补充回答
4. **多提醒场景下精准操作**：同时三个提醒触发时，"跳舞那个推迟一小时，其余我知道了" → 只推迟跳舞，其余结束

**一个关键设计**：唤醒词"小鹏小鹏"——因为系统在播报语音，用户也可能在说话，没有唤醒词会出现大量意外打断。参考 Siri（"Hey Siri"）、小爱同学（"小爱同学"）的交互逻辑。

### 4.4 P3：已有日程变更与周期控制（1.5min）

**为什么要单独一个 Proposal**：日程不是静态的——会议室坏了、临时有事、取消某次例会……这些变更如果每次都要手动进 App 改，语音闭环就不成立。

**变更能力一览**：

| 操作 | 示例 | 确认策略 |
| --- | --- | --- |
| 修改单次日程 | "明天下午的会改到四点半" | 无冲突直接执行 |
| 修改周期实例（仅本次） | "本周五的跑步改到周六" | 无冲突直接执行 |
| 修改周期事项（本次及以后） | "以后跑团都改到周六" | 高风险，需显式确认 |
| 跳过（仅本次） | "周五跑步不去了" | 单次日程等同取消 |
| 删除单次日程 | "把后天的会议删掉" | 显式确认 |
| 删除整个周期系列 | "删掉每周一早会这个系列" | 高风险，必须显式确认 |
| 撤销 | "哎不对，撤销刚才的修改" | 15min 内有效，跨会话也计入 |

**设计要点**：当改到与已有日程冲突时，复用 P1 的冲突处理逻辑；当存在多个匹配候选时，系统反问锁定唯一目标后再执行。

## 五、产品方案：关键决策与取舍（3min）

### 5.1 三个全局性决策（1.5min）

**决策一：语音优先，不是语音唯一**

语音是主交互通道，但 IM 作为关键操作的**回执**——修改成功/失败、日程确认，通过 IM 给一个可回看、可追溯的记录。语音负责快，IM 负责稳。

**决策二：先覆盖时间触发，不做位置/场景触发**

时间为日程的基本必要字段，位置等其他触发维度需要更多硬件能力和用户行为数据，属于后续迭代方向。一句话：把最重要的 80% 做好，不因 20% 拖慢节奏。

**决策三：App 始终前台运行的假设**

本期不处理后台/锁屏唤醒，专注在应用内完成完整语音闭环。后台唤醒涉及系统级能力，独立成后续 Proposal。

### 5.2 明确不做什么（1min）

（快速过表，点到为止）

| 不做 | 为什么 |
| --- | --- |
| 位置/蓝牙等非时间触发 | 额外硬件依赖，非 MVP 核心 |
| 方言及多语言识别 | 先保证普通话准确率 |
| 用户自定义唤醒词 | 增加误唤醒风险调试成本 |
| 复杂周期（工作日/节假日等） | 简单"每天/每周/每月"先跑通 |
| 跨日历源同步 | 独立产品能力，非闭环依赖 |
| 批量操作 | 一次一个，精准比效率优先 |

### 5.3 为什么是这个范围（30s）

> 8 周要交付的是一个**可用的闭环**，不是一个功能清单。我们选择的策略是：把一环做到 90 分，而不是把三环摊开到 60 分。一个用户从"记一件事"到"被提醒"到"随口回应"全流程跑通了，才能验证"语音优先"这个核心假设是否成立。

## 附：常见提问与应答预案

| 可能的问题 | 应答方向 |
| --- | --- |
| "这和 Siri 有什么区别？" | Siri 在创建提醒后，提醒响应仍靠 GUI；VoiceLife 把创建、提醒、修改三个动作都用语音闭环 |
| "唤醒词会不会误触发？" | 固定唤醒词 + 短音节词（嗯/哦）不做触发，降低误识别风险；自定义唤醒词后续扩展 |
| "语音被环境噪音干扰怎么办？" | 当前假设靠近麦克风使用，环境噪音消噪的工程专题独立成后续迭代 |
| "为什么不先做后台提醒？" | 前台是语音交互的前提，系统级唤醒涉及平台适配，MVP 验证闭环后再扩展 |
| "数据谁在管、隐私怎么保证？" | MVP 阶段本地存储优先，不上传云端；后续如需云端能力，在对应 Proposal 中明确隐私方案 |
