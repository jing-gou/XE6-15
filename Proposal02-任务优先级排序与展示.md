# Proposal 02：任务优先级排序与展示

## 动机 / 用户故事

我把 Repository 里的 Issue 和 PR 都导入进来了。问题是我对着一个长长的列表，还是不知道从哪下手。

现在我的办法是打开 GitHub 按时间倒序扫一遍，脑子里记"这个比较急、那个可以放放"——但新 Issue 一进来、Deadline 一改、被人 Request Review，脑子里的优先级马上乱掉。我也试过让 AI 帮我排序，但它给的顺序我看不懂，不敢信。

**说到底，我要的是一个能告诉我"先做什么"、并且能解释"为什么"的列表。** 排序依据必须是我能看懂、能核对的——不能是黑盒权重，也不能是没有数据就瞎编的 AI。

---

## 目标用户

- 已通过 Proposal 01 导入任务池的训练营成员；
- 手头同时挂着好几个待处理的 Issue 或 PR；
- Issue 、PR 状态经常更新。
- 想快速找到下一项该做的事，但也需要理解为什么是这一项排前面。

---

## 现在的做法及其不足

| 做法 | 问题 |
|------|------|
| GitHub 列表按创建时间倒序看 | 最新 ≠ 最紧急 |
| 自己在脑子里记优先级 | 撑不了几天，信息全在脑子里，一乱就忘 |
| AI 给我一个不可解释的顺序 | 不敢信，也不知道它是不是漏了什么 |

---

## 本期范围与明确不做

### 做

- [ ] 我能给每个事项设 P0/P1/P2 优先级，设完之后列表按优先级分组展示
- [ ] 排序我看得懂：先按优先级分组 → 同组里按 DDL 排 → 同 DDL 里按预估用时排
- [ ] 明天就到 DDL 的事项直接置顶，不管它是什么优先级
- [ ] 没设 DDL 的事项放久了会自动往上浮，防止被我忘掉
- [ ] 我可以拖拽事项手动调顺序，拖完之后我下次点排序时不会丢位置
- [ ] 卡片上能看到：标题、优先级标签、DDL、预估耗时、已耗时
- [ ] 把事项挪到"明日"时，弹窗问我"预估还需要多长时间"
- [ ] 点开事项能看到排序理由：为什么它排在当前这个位置
- [ ] Issue、PR、别人要我 Review 的 PR 全部混在一个列表里，共用一套排序规则

### 明确不做

- [ ] 不做偏离预警和到期提醒（另起 Proposal）
- [ ] 不做自然语言交互——"帮我排优先级"这种对话，这期只做手动设定界面（另起 Proposal）
- [ ] 不做 GitHub 数据同步引擎（属于基础设施，不在此 Proposal 范围内）
- [ ] 不做时间轴预览 Tab（MVP 先用列表视图）
- [ ] 不做四象限 / 矩阵视图（MVP 先用列表视图）
- [ ] 不做自定义排序规则的图形配置（排序规则这期是预设的，以后可以考虑放开）

## 关键决策与依据

| 决策 | 方案 | 结果及理由 |
|--------|---------|------------|
| **优先级设在哪里** | A. 直接读 GitHub 的 P0/P1/P2 Label | 不采纳——我们仓库的 wiki 没定义优先级标签体系，读不到 |
| | B. 我在 App 里自己设 | 采纳。优先级是我个人的判断，跟预估耗时、自定义 DDL 一样属于我自己的时间管理数据，不写回 GitHub |
| **排序怎么排** | A. 加权公式 + 硬规则覆盖 | 不采纳——权重数字我看不懂也没法验证，有硬规则说明公式自己在制造错误 |
| | B. 一层层往下排：优先级 → DDL → 预估用时 | 采纳。每一层我都能在脑子里还原，不需要额外规则兜底 |
| **没设 DDL 的事项怎么处理** | A. 紧急分归零，纯按优先级排 | 不采纳——P0 但没 DDL 的事会被有 DDL 的 P1 压到很下面，然后就忘了 |
| | B. 放久了自动往上浮 | 采纳。超过 7 天没动静的事项慢慢浮上来，不会一直被压在底下 |
| **PR 和 Review 怎么排** | A. PR/Review 单独分一组，各自排序 | 不采纳——我要的是"现在该做什么"这一个答案，分三组等于让我自己合并 |
| | B. Issue、PR、Review 全混在一起排 | 采纳。一个列表看所有事，PR 合并了自动沉到底部"已完成" |
| **我拖拽过的事项怎么办** | A. 拖过的就不参与排序了 | 不采纳——新事进来时被我拖过的事不再重排，反而要我手动维护 |
| | B. 拖过的保留相对位置，新事自动插入 | 采纳。我拖好的顺序不丢，新来的事按规则插到合适位置 |
| **是否考虑AI排序** | A. AI作为排序前置层，总管排序 | 不采纳——AI上下文不足，且存在温度差异问题，无法保证每一次的排序结果都是可控合理的。|
| | B. AI 作为后置层，参与排序结果的调整 | 采纳。可以确保每次的排序结果可控确定，AI仅作为结果调整参考 |

## 基本概念与信息结构

### 概念定义

| 概念 | 定义 | 理解 |
|------|------|------|
| 优先级 | P0 / P1 / P2 三级，我自己设 | P0 = 卡住或核心功能，P1 = 重要但能商量，P2 = 锦上添花、不紧急 |
| 排序层级 | 一层层往下筛：优先级 → DDL → 预估用时 | 我看一眼就知道顺序怎么来的："先按重要性分组，同组里看谁更急" |
| 紧急兜底 | DDL 在 24 小时内的事直接置顶 | 明天就要交的东西，不能被埋在任何分组里 |
| 老化因子 | 没设 DDL 的事项，放越久越往上浮 | 7 天开始慢慢上浮，防止被我彻底忘掉 |
| 变更标记 | 点排序后，位置变了的事项旁边有个 `🔄` | 让我一眼看到"上次排序之后哪些事挪了位置" |
| 今日事项 Tab | App 打开后的第一个页面，全部待办事都在这里 | 不按日期过滤——所有没做完的 Issue/PR 都参与排序。"今日"是说我现在看的这个视图 |
| PR Review 请求 | 别人要我 Review 的 PR | 跟我的 Issue、我的 PR 混在一起排，不单独分一组；放久了也会上浮 |

### 排序怎么排的——我看到的顺序是怎么来的

我点下排序按钮后，系统一层层往下筛，每一层都看得懂：

```
第一步 — 紧急的先拎出来
  明天就到 DDL 的事直接置顶，不参与后面的分层
  → 紧急事项互相之间还是按 DDL 近的排前面

第二步 — 按优先级分大组
  P0 一组、P1 一组、P2 一组，P0 永远在 P1 上面，P1 永远在 P2 上面
  没设优先级的：归到 P2 组末尾
  → 我第一眼看到的就是按重要性分的大框架

  同组里面进第三步：

  第三步 — 按 DDL 排
    有 DDL 的：离截止日期越近越靠前
    没 DDL 的：按"放了多久"慢慢往上浮
      放了不到 7 天 → 不动
      放了 7-14 天   → 稍微上浮
      放了 14-21 天  → 明显上浮
      放了 21 天以上 → 大幅上浮

    同 DDL（或同样放了差不多的天数）里面进第四步：

    第四步 — 按预估用时排
      用时短的在前面（能快速搞定的先做）
      没填预估用时的：排同档最后
```

### 排序依据展示

点开任意一个事项，我能看到排序理由，一层层解释它是怎么排到这个位置的：

```
"这件事排在前面是因为：
 1. 优先级 P0，高于其他 P1 事项
 2. 同 P0 组里 DDL 最近（3 天后）
 3. 预估只用 1 小时，比同组的其他事更快搞定
```

每条理由对应排序算法的一层，我可以自己对一遍：P0 是不是真的比 P1 高、DDL 是不是 3 天后等。

### 用户拖拽覆盖

- 我手动拖拽过的相对位置被记住
- 我下次点排序时，没拖过的事按系统建议重排，拖过的事保持我设的位置
- 新导入或数据变更的事按排序规则自动插到合适位置，不冲乱我拖好的布局
- 已完成的事自动沉到底部的"已完成"分区


## 验收标准

1. 我设好 P0/P1/P2 后，列表按优先级正确分组——P0 在上、P2 在下。
2. (即将)逾期的排在未逾期的前面，DDL 更近的更靠前。
3. 点开任意事项，都能看到它为什么排在当前位置。
4. 没设 DDL 的任务正常显示，带 ♾️ 标记。
5. 没填预估耗时的显示”待估算”，系统不替我瞎编数字。
6. 同样条件下顺序稳定——每次刷新不随机乱跳。
7. 排序只影响列表展示，不动我的 GitHub 数据、不自动改 DDL、不改计划日期。
8. 排序结果可以被 Proposal 03 读取并用于重排。
9. Demo 能跑通：导入四种时间状态的任务 → 点排序得到顺序 → 查看每项的排序理由。
10. 别人要我 Review 的 PR 跟我的 Issue、我的 PR 在同一个列表里混排。
