# Proposal 02：任务优先级排序与展示

## 动机 / 用户故事

训练营学生即使已经导入所有相关 Issue 和 PR，仍然需要逐项检查截止日期和状态，才能判断先做什么。单纯展示任务池只是把信息放到一个页面，没有减少优先级判断负担。

多个真实场景的共同点是：用户需要一个可解释的任务顺序，而不是一份无差别列表，也不能接受 AI 在没有依据时制造优先级。

**共同需求一句话提炼**：根据明确、可解释的信号对个人任务排序，并让用户知道为什么某项更优先。

---

## 目标用户

- 已通过 Proposal 01 导入单一 Repository 任务池的训练营成员；
- 同时存在多个待处理 Issue 或 PR；
- 希望快速找到下一项任务，但仍需要理解排序依据。

---

## 现有做法及其不足

| 做法 | 不足 |
|------|------|
| GitHub Issue 列表按创建时间倒序看 | 最新 ≠ 最紧急 |
| 自己在本地/脑子里记优先级 | 坚持几天就放弃，信息分散 |
| AI 给出一个不可解释的顺序 | 用户无法判断是否可信，也无法发现缺失数据 |

## 本期范围与明确不做

### 做

- [ ] 优先级体系：用户可在 App 侧对 Proposal 01 的个人任务池计算整体优先级排序和展示
- [ ] 排序算法：基于自定优先级、DDL 紧迫度、完成度、Issue依赖关系（目前仅考虑Sub-Issue关系）进行加权排序，展现为加权优先级
- [ ] 硬规则覆盖：P0 始终排在 P1 之前；DDL < 3 天自动升档；完成度 > 80% 加权提前
- [ ] 无限期 DDL 事项的老化衰减：超过 7 天未活动的事项随时间缓慢上浮
- [ ] 用户可手动拖拽覆盖系统排序，且拖拽后的相对位置在后续重排中被保留
- [ ] 每个事项卡片展示：标题、优先级标签、DDL、预估耗时、完成度
- [ ] 排序依据：排序后在详情页展示排序依据，让用户知道为什么这样排

### 明确不做

- [ ] 不做时间预估与耗时追踪（另起 Proposal）
- [ ] 不做偏离预警与提醒（另起 Proposal）
- [ ] 不做自然语言 Agent 的完整实现（仅定义优先级/DDL 设定接口，Agent 交互另起 Proposal）
- [ ] 不做 GitHub 数据同步引擎（属于基础设施，不在此 Proposal 范围内）
- [ ] 不做时间轴预览 Tab（属于视角一，不在 MVP 重点内）
- [ ] 不做四象限 / 矩阵视图（MVP 优先列表视图）
- [ ] 不做自定义排序规则的图形配置（排序规则 MVP 阶段由系统预设）
- [ ] 暂时不做Review事件，仅做Issue事件。

## 关键决策与依据

| 决策点 | 备选方案 | 结论与理由 |
|--------|---------|------------|
| **优先级存在哪里** | A. 从 GitHub Label 读取 P0/P1/P2 | 不采纳：wiki 规范未定义优先级标签体系 |
| | B. 用户在 App 侧设定 | 采纳：作为 App 侧时间管理属性，与预估耗时、自定义 DDL 同级，不与 GitHub 耦合 |
| **排序维度** | A. 仅基于 DDL | 不采纳：忽略任务重要性 |
| | B. DDL 优先，优先级  + 完成度 + 老化因子 加权 | 采纳：覆盖多维度，且每条数据源都在 App 侧可控 |
| **硬规则 vs 纯分数** | A. 纯加权分数排序 | 不采纳：用户看到 P0 排在 P1 下面会觉得系统有 bug |
| | B. 加权分数 + 硬规则覆盖 | 采纳：分数保底层逻辑，硬规则保直觉正确 |
| **无 DDL 事项处理** | A. 紧急分归零，按优先级纯排 | 不采纳：P0 的无 DDL 事项在 DDL 很紧的 P1 下面，容易被遗忘 |
| | B. 老化衰减 | 采纳：超过 7 天未活动的无 DDL Issue 随时间缓慢上浮 |
| **重排时机** | A. 用户手动触发 | 不采纳：偏离"自动化"价值主张 |
| | B. 事件驱动自动重排 | 采纳：变更即触发，排序始终反映最新状态 |
| **拖拽覆盖策略** | A. 拖拽后系统不再重排该事项 | 不采纳：新变更到来时被用户拖过的事项不再参与排序，用户反而要手动维护 |
| | B. 拖拽后保留相对位置，新变更插入 | 采纳：用户拖过的相对顺序不丢失，新增/变更的事项自动插入适当位置 |

## 基本概念与信息结构

### 概念定义

| 概念 | 定义 | 说明 |
|------|------|------|
| 优先级 | P0 / P1 / P2 三级 | P0 = 阻断性或核心功能，P1 = 重要但可协商，P2 = 锦上添花 |
| 排序权重 | 系统根据多维度计算出的数值，决定事项在列表中的位置 | 用户不可见，由排序引擎内部维护 |
| 硬规则 | 不依赖分数计算、直接改写排序结果的规则 | P0优先 / DDL < 3天升档 / 完成度 > 80%加权 |
| 老化因子 | 无 DDL 事项根据距最后活跃天数增加的隐性权重 | 7天开始累计，防止无限期 DDL 事项被遗忘 |
| 变更标记 | 事项因重排而发生位置变化时的视觉提示 | 如 `🔄` 标记，让用户感知系统自动做了什么 |
| 今日事项 Tab | 应用首屏，按排序结果展示当日待办列表 | 用户可在此设定优先级、DDL、拖拽调整 |

### 排序算法（MVP 版本）

```
基础分 = 紧急分 × 0.4 + 重要分 × 0.4 + 收尾分 × 0.2

紧急分 = f(距 DDL 天数)
  0-1 天 → 100
  2-3 天 → 80
  4-7 天 → 50
  8-14 天 → 20
  14+ 天 → 0
  无 DDL → 0 + 老化因子

重要分 = f(优先级)
  P0 → 100
  P1 → 60
  P2 → 20

收尾分 = f(完成度)
  80-100% → 100
  50-80%  → 50
  < 50%   → 0

老化因子 = f(距最后活跃天数)
  < 3 天  → 0
  3-7 天  → 5
  7-14 天 → 15
  14+ 天  → 30
```

### 硬规则（覆盖排序结果）

```
1. P0 的 Issue 始终排在所有 P1 的 Issue 前面
   — 例外：该 P0 的 DDL > 14 天且该 P1 的 DDL < 3 天

2. DDL 在 24 小时内的 Issue 自动置顶（无论优先级）

3. 用户手动拖拽过的顺序被记住
   — 新事项加入或变更时，插入到用户设定的相对位置
   — 用户未拖拽的事项始终按系统排序

4. 已完成的事项自动移到列表底部的"已完成"分区
```
## 验收标准

1. 导入任务能够按硬截止进入正确优先级分组。
2. 已逾期任务排在未逾期任务之前，更早硬截止具有更高优先级。
3. 每项任务都能展示明确的排序理由。
4. 无硬截止任务正常展示，并标记“♾️”。
5. 无预计投入任务标记“待估算”，系统不虚构数值。
6. 同等已确认条件下保持稳定顺序，不因刷新随机变化。
7. 排序过程不改变计划日期、今日任务、GitHub 状态或产品侧规划状态。
8. 排序结果可以被 Proposal 03 读取并用于重排。
9. Demo 能完成“导入四种时间状态的任务—得到顺序—查看每项排序理由”。
