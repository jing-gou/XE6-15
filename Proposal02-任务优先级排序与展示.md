# Proposal 02：任务优先级排序与展示

## 动机 / 用户故事

训练营学生即使已经导入所有相关 Issue 和 PR，仍然需要逐项检查截止日期和状态，才能判断先做什么。单纯展示任务池只是把信息放到一个页面，没有减少优先级判断负担。

多个真实场景的共同点是：用户需要一个可解释的任务顺序，而不是一份无差别列表，也不能接受 AI 在没有依据时制造优先级。

**共同需求一句话提炼**：根据明确、可解释的信号对个人任务排序，并让用户知道为什么某项更优先。

---

## 目标用户

- 已通过 Proposal 01 导入单一 Repository 任务池的训练营成员；
- 同时存在多个待处理 Issue 或 PR；
- 希望快速找到下一项任务，但仍需要理解排序依据。

---

## 现有做法及其不足

| 做法 | 不足 |
|------|------|
| GitHub Issue 列表按创建时间倒序看 | 最新 ≠ 最紧急 |
| 自己在本地/脑子里记优先级 | 坚持几天就放弃，信息分散 |
| AI 给出一个不可解释的顺序 | 用户无法判断是否可信，也无法发现缺失数据 |

## 本期范围与明确不做

### 做

- [ ] 优先级体系：用户可在 App 侧对 Proposal 01 的个人任务池计算整体优先级排序和展示
- [ ] 排序算法：按「优先级分组 → 同组内按 DDL 排序 → 同 DDL 内按完成度降序」逐层决策，排序依据对用户可解释
- [ ] 紧急兜底规则：DDL 在 24 小时内的 Issue/PR 自动置顶
- [ ] 无限期 DDL 事项的老化衰减：超过 7 天未活动的事项随时间缓慢上浮
- [ ] 用户可手动拖拽覆盖系统排序建议，且拖拽后的相对位置在用户下次触发排序时被保留
- [ ] 每个事项卡片展示：标题、优先级标签、DDL、预估耗时、完成度
- [ ] 用户将事项移至"明日"时，弹窗提示输入"预估还需用时"
- [ ] 排序依据：排序后在详情页展示排序依据，让用户知道为什么这样排
- [ ] 事件类型覆盖：Issue、PR、PR Review 三种事件类型纳入统一任务池，共用同一套排序规则

### 明确不做

- [ ] 不做自动耗时追踪（另起 Proposal）
- [ ] 不做偏离预警与提醒（另起 Proposal）
- [ ] 不做自然语言 Agent 的完整实现（仅定义优先级/DDL 设定接口，Agent 交互另起 Proposal）
- [ ] 不做 GitHub 数据同步引擎（属于基础设施，不在此 Proposal 范围内）
- [ ] 不做时间轴预览 Tab（属于视角一，不在 MVP 重点内）
- [ ] 不做四象限 / 矩阵视图（MVP 优先列表视图）
- [ ] 不做自定义排序规则的图形配置（排序规则 MVP 阶段由系统预设）

## 关键决策与依据

| 决策点 | 备选方案 | 结论与理由 |
|--------|---------|------------|
| **优先级存在哪里** | A. 从 GitHub Label 读取 P0/P1/P2 | 不采纳：wiki 规范未定义优先级标签体系 |
| | B. 用户在 App 侧设定 | 采纳：作为 App 侧时间管理属性，与预估耗时、自定义 DDL 同级，不与 GitHub 耦合 |
| **排序方式** | A. 加权公式 + 硬规则覆盖 | 不采纳：权重值缺少校准依据，硬规则的存在说明公式在制造错误，用户无法做心算验证 |
| | B. 层级排序：优先级分组 → 同组内按 DDL → 同 DDL 按完成度 | 采纳：每层决策独立可解释，不需要额外硬规则兜底，用户也能在脑中复现排序逻辑 |
| **无 DDL 事项处理** | A. 紧急分归零，按优先级纯排 | 不采纳：P0 的无 DDL 事项在 DDL 很紧的 P1 下面，容易被遗忘 |
| | B. 老化衰减 | 采纳：超过 7 天未活动的无 DDL Issue/PR 随时间缓慢上浮 |
| **重排时机** | A. 事件驱动自动重排 | 不采纳：排序仅为建议，自动重排等于替用户做决定 |
| | B. 用户手动触发排序 | 采纳：用户需要时点击排序，系统给出最新建议顺序但不自动改动列表 |
| **PR / Review 与 Issue 的关系** | A. PR/Review 独立分组，单独排序 | 不采纳：用户面对的是同一份"下一步做什么"的决策，分两组反而需要用户自己合并 |
| | B. Issue、PR、Review 混排，共用排序规则 | 采纳：统一任务池，用户看到的是一个排好序的列表；PR 被 merge 后自动归入已完成 |
| **拖拽覆盖策略** | A. 拖拽后系统不再重排该事项 | 不采纳：新变更到来时被用户拖过的事项不再参与排序，用户反而要手动维护 |
| | B. 拖拽后保留相对位置，新变更插入 | 采纳：用户拖过的相对顺序不丢失，新增/变更的事项按排序规则列入合适位置 |

## 基本概念与信息结构

### 概念定义

| 概念 | 定义 | 说明 |
|------|------|------|
| 优先级 | P0 / P1 / P2 三级 | P0 = 阻断性或核心功能，P1 = 重要但可协商，P2 = 锦上添花 |
| 排序层级 | 排序按「优先级 → DDL → 完成度」逐层应用，上层决定分组顺序 | 用户可见，排序依据可逐层展示："先按优先级，同优先级里按截止日期" |
| 紧急兜底 | DDL 在 24 小时内的 Issue/PR 直接置顶，不参与层级排序 | 防止极端场景下紧急事项被埋没 |
| 老化因子 | 无 DDL 事项根据距最后活跃天数增加的隐性权重 | 7天开始累计，防止无限期 DDL 事项被遗忘 |
| 变更标记 | 用户触发排序后，事项位置相比上次发生变化时的视觉提示 | 如 `🔄` 标记，让用户感知排序建议发生了什么变化 |
| 今日事项 Tab | 应用首屏，按排序结果展示全部待办列表（不按日期筛选，所有未完成 Issue/PR 均参与排序） | 用户可在此设定优先级、DDL、拖拽调整；"今日"指当前视图，非日期过滤 |
| PR Review 请求 | 用户作为 reviewer 被请求 review 的 PR | 与 Issue、自有 PR 共用同一套排序规则；Review 请求的 DDL 可从 PR 创建时间或 requested_at 计算老化 |

### 完成度计算

完成度 = 已用时 / (预估用时 + 预估还需用时)

- **预估用时**：用户创建/编辑事项时手动输入的初始估算
- **预估还需用时**：用户将事项移至"明日"时弹窗提示输入，无需在卡片上常驻展示
- **已用时**：用户手动输入（MVP 阶段不做自动计时）
- 未输入预估用时或预估还需用时的事项：完成度为"空"，排序时排在同等条件有完成度的事项之后，按导入顺序稳定排列
- 分母（预估用时 + 预估还需用时）为 0 时：完成度视为"空"

### 排序算法（MVP 版本）

排序引擎按层级决策，每层独立排序后进入下一层：

```
Step 0 — 紧急拦截（最先执行）
  DDL 在 24 小时内的 Issue/PR 直接置顶，跳过后续所有层级
  → 紧急事项彼此之间仍按 DDL 近→远排列

Step 1 — 优先级分组
  P0、P1、P2 各自成组，P0 永远在 P1 之上，P1 永远在 P2 之上
  未设定优先级的事项：默认归入 P2 组末尾
  → 用户看到的第一条就是：先按重要性分组

  同组内进入 Step 2：

  Step 2 — 按 DDL 排序
    有 DDL：距 DDL 越近越靠前
    无 DDL：按老化因子计算虚拟紧迫度（仅对无 DDL 事项生效）
      < 7 天  → 不升档
      7-14 天 → 轻微上浮
      14-21 天→ 明显上浮
      21+ 天  → 大幅上浮

    同 DDL（或同老化档位）内进入 Step 3：

    Step 3 — 按预估用时排序（短 → 长）
      未填预估用时：排在同档末尾

      同预估用时内进入 Step 4：

      Step 4 — 按完成度降序
        完成度高的排在前面（快收尾的先做）
        完成度相同或都为空：按首次导入顺序（imported_at）稳定排列
```

### 排序依据展示

排序依据对用户逐层展示，例如：

```
"此事项排在前面是因为：
 1. 优先级 P0 > 其他 P1 事项
 2. 同 P0 组内 DDL 最近（3 天后）
 3. 完成度 85%，快收尾了"
```

### 用户拖拽覆盖

- 用户手动拖拽过的相对位置被记录
- 用户下次触发排序时，未拖拽的事项按系统排序建议排列，已拖拽的事项保持在用户设定的相对位置
- 新增 / 变更的事项按排序规则列入合适位置，不破坏已有拖拽布局
- 已完成的事项自动下沉到列表底部"已完成"分区


## 验收标准

1. 导入任务的优先级分组正确反映用户设定的 P0/P1/P2 级别。
2. 已逾期任务排在未逾期任务之前，更早硬截止具有更高优先级。
3. 每项任务都能展示明确的排序理由。
4. 无硬截止任务正常展示，并标记“♾️”。
5. 无预估耗时任务标记”待估算”，系统不虚构数值。
6. 同等已确认条件下保持稳定顺序，不因刷新随机变化。
7. 排序过程不改变计划日期、今日任务、GitHub 状态或产品侧规划状态。
8. 排序结果可以被 Proposal 03 读取并用于重排。
9. Demo 能完成“导入四种时间状态的任务—得到顺序—查看每项排序理由”。
10. Issue 关闭或 PR 合并后，在排序列表中同步后自动归入"已完成"分区。
11. 用户将事项移至"明日"时，弹窗提示输入预估还需用时，输入后参与完成度计算。
12. PR Review 请求与 Issue、自有 PR 在同一列表中混排，共用排序规则。
