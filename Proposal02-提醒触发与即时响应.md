# P2：提醒触发与即时响应

> 状态：Draft / FullSpec  
> 单一设计点：定义提醒如何到达用户，以及用户在收到提醒后如何在无 GUI 下完成响应。  
> 依赖：P1（语音创建与查询）。  
> 本 Proposal 不修改底层日程数据，不改变周期规则，不涉及已有日程的变更流程。

## 动机 / 用户故事

作为一名有零散日程和周期提醒需求的个人用户，我用语音创建了提醒（P1），但到了提醒响起的时刻，我可能正在开会、走路、做饭或通勤。

现在遇到的麻烦是：手机响了，我看了一眼通知，然后就没然后了——等我忙完已经忘了刚才那条通知说的是什么。我也试过打开日历 App 去处理，但只是「推迟 5 分钟」这个动作，就要解锁、找到那条提醒、选推迟、再输时间——等做完这些，我已经被打断了。

如果提醒响起时我能直接说一句「知道了」「十分钟后再提醒」或者「先忽略」，而不需要解锁手机、打开 App、找到条目再操作，那我可以在不中断当前事的前提下，把这条提醒处理掉。

而如果我把这条提醒的内容听错了、或者想改个时间，提醒场景也应该能平滑衔接到日程变更流程——但我先说清楚：响应提醒时「只处理提醒本身」和「去改日程」是两件不同的事。

**共同需求一句话提炼**：提醒到达时，用户正在忙；用户只需一句话就能处理提醒状态，且这个处理不改变底层日程。

## 目标用户

- 已通过 P1 建立日程和提醒的个人用户；
- 经常在提醒响起时处于不方便操作屏幕的场景（开会、走路、做饭、通勤、照顾家人）；
- 希望通过一句话快速处理当前提醒（知道了 / 稍后 / 忽略 / 标记已处理），而非打开 App；
- 「处理提醒」和「修改日程」是两件事——前者只改提醒状态，后者交给 P3。

## 现有做法及其不足

| 用户当前怎么做 | 仍然存在的问题 |
|---|---|
| 手机系统闹钟或日历通知 | 闹钟响了，按掉就没了；用户忙完后不记得刚才提醒的是什么 |
| 打开日历 App 手动推迟 | 推迟 5 分钟就要解锁、找条目、选推迟——打断成本高 |
| 看到推送通知后无视 | 当时没法处理，等忙完已经忘了通知内容 |
| 智能音箱语音提醒 | 只能听，不能回；操作还得掏出手机 |

## 本期范围与明确不做

### 本期范围

#### 1. 提醒通过语音和 IM 同时触发

P1 创建的日程到达提醒时间后，提醒通过语音和 IM（如企业微信、钉钉、飞书）同时推送给用户。用户不一定需要打开 App，语音播报和 IM 消息都能收到提醒。

#### 2. 用户可通过一句话完成即时响应

用户在收到提醒后，无需 GUI 操作，通过语音一句话完成响应。响应命令限定为 4 种，只改变提醒状态，不改变底层日程：

- 「收到了」→ 标记为已读，不再重复播报
- 「稍后提醒我」/「X 分钟后提醒我」→ 推迟提醒，重新设定触发时间
- 「忽略本次」→ 忽略本次提醒（如果是周期事项，不影响下一次）
- 「已处理」/「标记已处理」→ 提醒标记为已处理

#### 3. 响应只改变提醒状态，不修改底层日程

用户对提醒的每一句话响应，都只影响提醒对象本身的状态。不修改日程的标题、时间、地点、周期规则等内容。如果用户需要改日程，系统告知用户进入 P3 流程。

#### 4. 简单提醒一句话说完，复杂提醒先摘要

提醒播报按内容复杂度分层：

- 简单提醒（一次性的短事项）：语音直接播完
- 复杂提醒（有地点、备注、或多个时间点的事项）：先播标题、时间、影响范围摘要，再问用户是否需要听详情；完整详情写入 IM 回执

#### 5. 每次响应留下 IM/文本回执

用户对提醒的每次响应——收到了、推迟了、忽略了、处理了——都生成一条 IM/文本回执。用户可以事后翻看「今天有哪些事提醒过我，我怎么处理的」。

#### 6. 提醒触发时如果用户意图转入 P3

用户说「把会议改到今晚八点」「今天这次不参加了」「停止后续提醒」时，系统不在此处执行，而是告知用户「这是日程变更操作，将转入修改流程」（P3），并在 P3 中以高风险规则确认影响范围。

### 本期明确不做

- 不通过提醒响应修改底层日程的标题、时间、地点或周期规则（属于 P3）；
- 不在提醒处理中做周期范围确认（本次 / 以后 / 整个系列属于 P3）；
- 不提供面向用户的日历 GUI；
- 不依赖位置、摄像头、传感器触发提醒；
- 不做全天候录音或被动记忆捕获；
- 不做多用户身份识别或家庭成员区分；
- 不定义语音入口的具体载体（智能音箱、耳机、车载等）；
- 不做外部信号监控（如「老张三点没回消息就提醒我」）；
- 不做跨天模糊记忆检索。

## 关键决策与依据

### 决策 1：提醒响应只改变提醒状态，不改底层日程

| 方案 | 用户体验 | 判断 |
|---|---|---|
| 用户说「十分钟后提醒」时直接把日程时间改到十分钟后 | 看似方便，但「改提醒时间」和「改日程时间」是两件事；用户可能只想推迟提醒，日程本身还是原来那个时间 | 混淆了提醒和日程的职责；用户无法预期一句话到底改了哪个 |
| 用户说「十分钟后提醒」只推迟此条提醒，日程内容不变 | 用户明确知道：「推迟提醒」不影响日程本身；日程时间改不改由 P3 控制 | 边界清晰，用户可预测效果 |

**选择**：提醒响应只操作提醒状态。用户说「稍后提醒我」后，日程的原始时间不变，只是这条提醒推迟了。如果用户想改日程时间，系统告知需要转入 P3 变更流程。

### 决策 2：提醒通过语音和 IM 双通道同时触发

| 方案 | 用户体验 | 判断 |
|---|---|---|
| 仅语音播报 | 语音听过即逝，用户忙完后无法回看；复杂事项靠耳朵记不住 | 语音适合当时处理，不适合事后确认 |
| 语音播报 + IM 消息回执 | 语音当时听，IM 事后查；用户可以在 IM 里看到这条提醒的完整内容和自己的响应记录 | 双通道互补，降低听过即逝的问题 |

**选择**：提醒同时通过语音播报和 IM 消息到达用户。语音负责即时通知，IM 负责持久化记录和回溯。

### 决策 3：按内容复杂度分层播报

| 方案 | 用户体验 | 判断 |
|---|---|---|
| 无论内容多少，全部语音播报 | 简单提醒还好，复杂提醒（多条备注、地点、反复确认）语音太长，用户等不及听完 | 语音过长会打断用户当前工作节奏 |
| 简单短播 + 复杂先摘要 | 简单提醒一句话播完；复杂提醒先播「什么事、几点、到几点」，详情写入 IM | 用户自己选：如果只是想知道「什么响」，摘要就够了；如果需要详情，去 IM 看 |

**选择**：按复杂度分层。系统根据提醒内容字段数量判断复杂度，简单提醒直接播报，复杂提醒先播摘要。

### 决策 4：有限命令集 + 自然语言「稍后」时间

| 方案 | 用户体验 | 判断 |
|---|---|---|
| 只允许固定短语 | 准确率高，但用户不一定会使用精确的固定短语 | 需要用户记忆规则，不自然 |
| 开放自然语言自由响应 | 用户可以说「十分钟后再叫我」「等一会儿再说」「先放着」，但意图解析依赖 AI，有不确定性 | 不限制自然语言表达方式，但最终映射到有限的几条动作 |

**选择**：命令不限固定短语，系统通过关键词匹配意图，最终映射到 4 种提醒操作（已读 / 推迟 / 忽略 / 已处理）。例如「十分钟后再叫我」→ 推迟 10 分钟，「先放着」→ 已读。匹配不到时，告知用户可用的操作类型。

### 决策 5：提醒触发时识别 P3 意图并引导跳转

| 方案 | 用户体验 | 判断 |
|---|---|---|
| 提醒场景下用户说「改到八点」，系统直接执行 | 方便，但用户可能只是想推迟提醒，系统却改了日程；后续无法追溯「为什么日程变了」 | 混淆 P2 和 P3 职责，用户无法预期 |
| 提醒场景下用户说「改到八点」，系统告知「这是修改日程操作」，进入 P3 流程 | 用户明确知道「我在改日程，不是推迟提醒」；P3 以高风险规则确认影响范围 | 职责清晰，操作可预测 |

**选择**：提醒场景中检测到「改时间」「跳过本次」「停止后续」等意图时，系统不在此处执行，而是告知将进入 P3 变更流程，并在 P3 中以高风险规则确认范围。

## 基本概念与信息结构

### 用户会看到的核心对象

| 概念 | 用户理解 |
|---|---|
| 提醒 | 到时间了系统告诉你「有一件事该做了」。一个日程可以触发一次或多次提醒（周期事项每周期一次）。 |
| 提醒状态 | 这条提醒当前处于什么状态：等待触发、已推送、已读、已推迟、已忽略、已处理。 |
| 即时响应 | 提醒响起后，不用打开 App，直接说一句话就算处理了。响应只影响提醒本身，不改变日程。 |
| 回执 | 每次提醒触发和每次用户响应，IM 里都有一条记录。用户过后可以翻看。 |
| P3 变更流程 | 如果用户想改日程本身（时间、地点、周期规则），不是简单处理提醒。系统会引导进入 P3。 |

### 提醒触发的时机

每一条由 P1 创建的日程到达其提醒时间（创建时用户指定的提前提醒时间或默认提前时间）时触发。

- 一次性事项：在指定的提醒时间触发
- 周期事项：每个周期实例的提醒时间均触发

### 用户可见的提醒状态

| 状态 | 用户看到什么 |
|---|---|
| 待触发 | 提醒已设定，尚在等待触发时间 |
| 已推送 | 提醒已通过语音和 IM 发出，等待用户响应 |
| 已读 | 用户回应了「收到了」，不再重复播报 |
| 已推迟 | 用户要求稍后提醒，系统将在指定时间后再次推送（可多次推迟） |
| 已忽略 | 用户说「忽略本次」，此条提醒不再推送；周期事项的下次提醒不受影响 |
| 已处理 | 用户回应「已处理」，提醒标记为完成 |

### 提醒内容包含什么

简单提醒：

> 「提醒：明天上午十点客户拜访」

复杂提醒先播摘要：

> 「提醒：明天上午十点客户拜访——A 座 1503——预计两小时」
> 「详情已发到你的 IM」

IM 回执中的完整内容：

```
📌 提醒：客户拜访
━━━━━━━━━━━━━━━━━
时间：明天（7/22）10:00 - 12:00
地点：A 座 1503
备注：带合同复印件、身份证
━━━━━━━━━━━━━━━━━
你的回复：[稍后提醒] 已推迟至 10:10
━━━━━━━━━━━━━━━━━
可回复：
「收到了」 → 标记已读
「X 分钟后提醒」 → 推迟
「忽略本次」 → 本次不提醒
「已处理」 → 标记完成
```

### P2 与 P3 的边界

```
用户收到提醒
       │
       ├── 用户说「收到了」「稍后」「忽略」「已处理」
       │   └── P2：只改提醒状态，不改日程
       │
       ├── 用户说「改到今晚八点」「今天跳过」「停止后续」
       │   └── P3：修改日程内容或周期规则（高风险确认）
       │
       └── 用户说无法识别的命令
           └── 系统回复可用操作提示
```

### 用户使用路径

```text
P1 创建的日程到达提醒时间
       ↓
系统通过语音 + IM 同时发出提醒
       ↓
用户正在忙（开会、走路、做饭、通勤）
       ↓
用户一句话响应
       ↓
   ┌── P2 命令 → 提醒状态更新；日程不变 → IM 回执记录
   └── P3 意图 → 告知用户转入 P3 变更流程
```

## 原型 / Demo

### 场景一：稍后提醒

**现状**

小林设置了「明天上午十点客户拜访」14:30 提醒。14:30 时他正在和同事开会，手机响了一声。

**提议后的体验**

14:30，小林手机语音播报：「提醒：明天上午十点客户拜访」。同时 IM 收到一条提醒消息。

小林不方便看屏幕，直接说了一句「十分钟后提醒我」。

系统回复（语音）：「好的，已推迟至 14:40」。

14:40，系统再次语音播报提醒。

**只发生了什么变化**

| 对象 | 变化 |
|---|---|
| 提醒状态 | 已推送 → 已推迟（稍后再次触发） |
| 日程「客户拜访」 | 不变（时间仍然是明天上午十点） |

### 场景二：已处理 + IM 回执

**现状**

小林早上收到「8:00 提醒吃药」。他正好刚吃完药。

**提议后的体验**

语音播报：「提醒：8:00 吃药」。小林说「已处理」。

系统回复（语音）：「好的，已标记为已处理」。同时 IM 推送一条回执：

```
📌 提醒：吃药
回复：[已处理]
━━━━━━━━━━━━━━━━━
如需查看今日所有提醒，请输入「今天提醒」
```

**只发生了什么变化**

| 对象 | 变化 |
|---|---|
| 提醒状态 | 已推送 → 已处理 |
| 日程「吃药」 | 不变（明天的周期实例仍然会在 8:00 触发） |

### 场景三：忽略本次（周期事项）

**现状**

小林有「每周五下午五点提交周报」的周期提醒。这周五他请假了，不需要交。

**提议后的体验**

周五 16:30 语音播报：「提醒：今天 17:00 提交周报」。小林说「忽略本次」。

系统回复（语音）：「好的，本次周报提醒已忽略，下次提醒仍然是下周五下午五点」。IM 回执同步确认。

**只发生了什么变化**

| 对象 | 变化 |
|---|---|
| 提醒状态（本周五） | 已推送 → 已忽略 |
| 提醒（下周五） | 不受影响，按原周期触发 |
| 周期事项「提交周报」 | 不变 |

### 边界与非法情况

1. 用户说「十分钟后提醒我」→ 系统推迟 10 分钟，回复「好的，已推迟至 HH:MM」。
2. 用户说「晚点再说」→ 未指定时间，系统默认推迟 10 分钟，回复「好的，10 分钟后再次提醒」。
3. 用户说「改到今晚八点」→ 检测为 P3 意图，回复「这是修改日程操作，将为你转入日程修改流程」。
4. 用户说「把周报改到周六」→ 检测为 P3 意图（修改周期实例），引导进入 P3。
5. 用户说「今天的吃药提醒取消」→ 检测为 P3 意图（跳过一次周期），引导进入 P3。
6. 用户说无法识别的命令 → 系统回复「无法识别，请说：收到了 / X 分钟后提醒 / 忽略本次 / 已处理」。
7. 已推到「已读」或「已处理」的提醒再次被推迟 → 系统回复「此提醒已处理」。

## 验收标准

1. P1 创建的日程到达提醒时间后，提醒同时通过语音和 IM 触发。
2. 用户说「收到了」后，提醒状态变为已读；底层日程不变。
3. 用户说「十分钟后提醒我」后，提醒推迟 10 分钟后再次触发；底层日程不变。
4. 用户说「忽略本次」后，本次提醒不再推送；周期事项的下次提醒不受影响。
5. 用户说「已处理」后，提醒状态变为已处理；底层日程不变。
6. 简单提醒（标题 + 时间）语音直接播完，不超 5 秒。
7. 复杂提醒（含地点、备注）语音先播摘要，详情在 IM 回执中展示。
8. 用户说「改到今晚八点」系统不执行修改，告知转入 P3 流程。
9. 用户说无法识别的命令，系统给出可用操作提示。
10. 每次提醒触发和用户响应均生成 IM/文本回执，包含原始提醒内容和响应记录。
11. 提醒状态变化不影响底层日程的任何字段。
12. Demo 能跑通：P1 创建日程 → 到达提醒时间 → 语音+IM 触发 → 用户一句话响应 → 确认提醒状态变更但日程不变。
