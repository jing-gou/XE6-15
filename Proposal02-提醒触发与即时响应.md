# P2：提醒触发与即时响应

> 状态：Draft / FullSpec  
> 单一设计点：定义提醒如何到达用户，以及用户在收到提醒后如何在无 GUI 下完成响应。  
> 依赖：P1（语音创建与查询）。P2 接收由 P1 创建、已包含明确触发时间的提醒；P2 不负责决定默认提前时间。  
> 本 Proposal 不修改底层日程数据，不改变周期规则，不涉及已有日程的变更流程。

## 动机 / 用户故事

作为一名有零散日程和周期提醒需求的个人用户，我用语音创建了提醒（P1），但到了提醒响起的时刻，我可能正在开会、走路、做饭或通勤。

现在遇到的麻烦是：手机响了，我看了一眼通知，然后就没然后了——等我忙完已经忘了刚才那条通知说的是什么。我也试过打开日历 App 去处理，但只是「推迟 5 分钟」这个动作，就要解锁、找到那条提醒、选推迟、再输时间——等做完这些，我已经被打断了。

如果提醒响起时我能直接说一句「知道了」「十分钟后提醒」或者「不用再响了」，而不需要解锁手机、打开 App、找到条目再操作，那我可以在不中断当前事的前提下，把这条提醒处理掉。

而如果我想改的是日程本身——把会议改个时间、今天不参加了——那是另一件事。提醒场景能分清：我只处理提醒，不改日程。

**共同需求一句话提炼**：提醒到达时，用户正在忙；用户只需一句话就能处理提醒状态，且这个处理不改变底层日程。

## 目标用户

- 已通过 P1 建立日程和提醒的个人用户；
- 经常在提醒响起时处于不方便操作屏幕的场景（开会、走路、做饭、通勤、照顾家人）；
- 希望通过一句话快速处理当前提醒，而非打开 App；
- 「处理提醒」和「修改日程」是两件事——前者只改提醒状态，后者交给 P3。

## 现有做法及其不足

| 用户当前怎么做 | 仍然存在的问题 |
|---|---|
| 手机系统闹钟或日历通知 | 闹钟响了，按掉就没了；用户忙完后不记得刚才提醒的是什么 |
| 打开日历 App 手动推迟 | 推迟 5 分钟就要解锁、找条目、选推迟——打断成本高 |
| 看到推送通知后无视 | 当时没法处理，等忙完已经忘了通知内容 |
| 智能音箱语音提醒 | 只能听，不能回；操作还得掏出手机 |

## 本期范围与明确不做

### 本期范围

#### 1. 提醒通过语音和 IM 通道同时触发

P1 创建的日程到达提醒时间后，提醒通过语音和 IM 通道同时推送给用户。语音和 IM 独立发送，一条通道失败不阻塞另一条。语音负责即时通知，IM 负责持久化记录和回执。

#### 2. 用户可通过一句话关闭或推迟当前提醒

用户收到提醒后，通过语音或 IM 回复一句话完成响应。响应只改变当前提醒的状态，不改变底层日程。P2 只保留两种核心提醒操作：

- **确认收到 / 关闭**：「知道了」「不用再响了」「关闭这条提醒」→ 当前提醒结束，底层日程不变
- **稍后提醒**：「十分钟后提醒」「稍后再说」→ 推迟当前提醒，等待再次触发

用户没有指定推迟时长时，系统应追问「多久后再提醒？」。只有用户预设置了默认推迟时长，才能不追问直接使用。

#### 3. 涉及日程变更的意图转入 P3

用户说「这次不做了」「今天不参加了」「这周跳过」「改到今晚八点」「停止后续提醒」时，系统告知用户「将转入日程修改或周期控制流程」（P3），以高风险规则确认范围。P2 不处理任何修改日程内容的操作。

#### 4. 语音默认播报标题和发生时间；完整内容进入 IM

语音默认播报标题和发生时间。地点、备注等完整内容写入 IM 回执。用户说「再详细说说」时继续播报剩余内容。

#### 5. 每次响应留下 IM 回执

用户对提醒的每次响应——关闭、推迟——都生成一条 IM 回执，包含原始提醒内容、用户响应和处理时间。用户可以事后翻看。

#### 6. 未响应和通道失败的处理

- 语音播报失败但 IM 成功：通知不以语音送达，但 IM 消息保留，视为已推送
- IM 发送失败但语音成功：语音正常播报，后续重试 IM
- 用户未在合理时间内回应：不重复语音播报（避免打扰），IM 记录保留以备用户事后查看。不安排自动重播
- 多条提醒同时到达：语音依次播报，IM 分别留存
- 用户连续推迟：同一提醒最多允许推迟 3 次，第 3 次推迟到期后标记为「已多次推迟」，不再重复播报
 - 推迟后的提醒重新进入等待触发，到达时间后再次语音和 IM 同时推送，并注明「这是之前推迟过的提醒」

#### 7. 并发提醒的活动提醒与响应绑定

当多条提醒同时到达时，语音依次播报。当前正在播报或最近播报完成的提醒为「活动提醒」，用户的语音响应默认作用于活动提醒。

- 用户在上一条播报结束后、下一条开始前说出响应：响应作用于当前已播完的提醒
- 用户在所有提醒播完后说出「知道了」：触发歧义确认，系统列举各提醒标题，请用户指定目标——「好的，请问你指的是「吃药」还是「出门开会」？」
- 用户在所有提醒播完后说出「十分钟后提醒」：歧义确认时需同时指定时间和目标
- IM 回执各自独立，用户在各自回执中的回复只作用于该回执对应的提醒，不产生歧义

### 本期明确不做

- 不通过提醒响应修改底层日程的标题、时间、地点或周期规则（属于 P3）；
- 不处理「跳过一次」「今天不参加」「停止后续」等周期实例变更（属于 P3）；
- 不将「已处理」作为提醒响应命令——事项完成状态不属于提醒工具职责；
- 不提供面向用户的日历 GUI；
- 不依赖位置、摄像头、传感器触发提醒；
- 不做全天候录音或被动记忆捕获；
- 不做多用户身份识别或家庭成员区分；
- 不定义语音入口的具体载体（智能音箱、耳机、车载等）；
- 不做外部信号监控（如「老张三点没回消息就提醒我」）；
- 不做跨天模糊记忆检索。

## 关键决策与依据

### 决策 1：提醒响应收敛为两种核心操作

| 方案 | 用户体验 | 判断 |
|---|---|---|
| 四种响应：收到了 / 稍后 / 忽略本次 / 已处理 | 用户容易混淆「忽略本次」和「已处理」与真实事项状态——「已处理」暗示事项已完成，但系统并不标记日程完成 | 名词语义与用户心理模型不匹配 |
| 两种响应：关闭 / 稍后 | 用户明确知道「我只是处理了这条提醒，事项本身没有变化」 | 边界清晰，不引入完成语义 |

**选择**：P2 只保留「关闭」（确认收到 / 不再播报）和「稍后提醒」两种操作。删去「忽略本次」和「已处理」。用户说「这次不做了」「今天不参加了」等涉及事项完成或跳过的意图，引导转入 P3。

### 决策 2：提醒通过语音和 IM 双通道独立发送

| 方案 | 用户体验 | 判断 |
|---|---|---|
| 仅语音播报 | 语音听过即逝，用户忙完后无法回看；复杂事项靠耳朵记不住 | 语音适合当时处理，不适合事后确认 |
| 语音播报 + IM 消息回执 | 语音当时听，IM 事后查；一条通道失败不阻塞另一条 | 双通道互补，降低听过即逝的问题 |

**选择**：提醒同时通过语音播报和 IM 消息到达用户。两条通道独立发送，相互不阻塞。

### 决策 3：语音默认播标题和时间，完整内容走 IM

| 方案 | 用户体验 | 判断 |
|---|---|---|
| 按字段数量判断复杂度，分层播报 | 字段多不一定语音内容复杂；分类规则用户不可感知 | 实现方法混入产品决策，不可验收 |
| 一律播标题和时间，详情走 IM | 规则统一，用户一听就知道「标题 + 几点」；想看详情去 IM | 用户可预期，规则简单可验收 |

**选择**：语音默认播报标题和发生时间。地点、备注等完整内容进入 IM 回执。用户要求听详情时继续播报。

### 决策 4：推迟时未指定时长应追问

| 方案 | 用户体验 | 判断 |
|---|---|---|
| 未指定时长时默认 10 分钟 | 「晚点」可能是半小时后，「等一会儿」可能是 5 分钟；用户意图与默认值可能不同 | 猜测用户意图，可能猜错 |
| 未指定时长时追问「多久后？」 | 用户多回答一次，但得到的是用户自己的时间 | 交互多一轮，但确定性强 |

**选择**：用户没有指定推迟时长时，系统追问「多久后再提醒？」。只有用户预设置了默认推迟时长，才不追问直接使用。

### 决策 5：IM 回执只绑定当前提醒，不做日历查询入口

| 方案 | 用户体验 | 判断 |
|---|---|---|
| IM 回执中提供「如需查看今日所有提醒，请输入今天提醒」 | 用户开始在 IM 中查询日历，IM 成为开放式日历入口 | 与战略决策「IM 不做独立日历管理入口」冲突 |
| IM 回执只记录和回复当前提醒 | 用户只能对当前提醒做受控回复（关闭 / 推迟）；查询由 P1 语音完成 | 与战略决策一致，职责清晰 |

**选择**：IM 回执只记录和处理当前提醒。查询已创建的日程通过 P1 语音查询完成。

### 决策 6：未响应不自动重播，IM 记录兜底

| 方案 | 用户体验 | 判断 |
|---|---|---|
| 未响应时自动重播，最多 3 次 | 如果用户正在开会或睡觉，连续播报会造成打扰 | 用户不能选择是否被打断 |
| 未响应时只推送一次，IM 记录保留 | 用户忙完后通过 IM 查看提醒记录，不会错过内容 | 一次播报 + IM 兜底，不重复打扰 |

 **选择**：提醒只主动推送一次。用户忙完后通过 IM 回执查看未响应的提醒。连续推迟最多 3 次，第 3 次后不再播报。

### 决策 7：IM 受控回复与语音响应具有同等效力

| 方案 | 用户体验 | 判断 |
|---|---|---|
| IM 只展示、不接收回复 | 用户只能在 IM 上查看，要操作还得切语音 | 语音是主入口，但用户可能处于不方便语音的环境（会议室、安静场合） |
| IM 接收受控回复（关闭/推迟），效力与语音相同 | 用户在安静场合可以通过 IM 完成同样的操作；IM 和语音的响应结果一致 | 两通道效力等同，用户按场景自选 |

**选择**：IM 受控回复和语音响应具有同等效力，都只改变提醒状态。IM 回复仅限「知道了」（关闭）和「X 分钟后提醒」（推迟），不支持其他操作。语音入口支持更灵活的时间表述，IM 入口通过固定按钮或简短文本匹配实现。

IM 回复的受控按钮由 IM 消息中的回执模板提供，用户在 IM 中点击或输入受控命令，效果等同于语音说出同一命令。

## 基本概念与信息结构

### 用户会看到的核心对象

| 概念 | 用户理解 |
|---|---|
| 提醒 | 到时间了系统告诉你「有一件事该做了」。一个日程可以触发一次或多次提醒（周期事项每周期一次）。 |
| 提醒状态 | 这条提醒当前处于什么状态：等待触发、已推送、已关闭、已推迟。 |
| 即时响应 | 提醒响起后，不用打开 App，直接说一句话就算处理了。响应只关闭或推迟这条提醒，不改变日程。 |
| 回执 | 每次提醒触发和每次用户响应，IM 里都有一条记录。用户过后可以翻看。 |
| P3 变更流程 | 如果用户想改日程本身（时间、地点、周期规则），不是简单处理提醒。系统会引导进入 P3。 |

### 提醒触发的时机

P1 创建的日程到达提醒时间时触发。P2 接收已包含明确触发时间的提醒，不负责决定默认提前多久。

- 一次性事项：在指定的提醒时间触发
- 周期事项：每个周期实例的提醒时间均触发

### 用户可见的提醒状态

| 状态 | 用户看到什么 |
|---|---|
| 待触发 | 提醒已设定，尚在等待触发时间 |
| 已推送 | 提醒已通过语音和 IM 发出，等待用户响应 |
| 已关闭 | 用户确认收到或要求关闭，不再播报 |
| 已推迟 | 用户要求稍后提醒，将在指定时间后再次推送；同一提醒最多 3 次 |
| 已多次推迟 | 同一提醒已推迟 3 次，不再播报；IM 保留记录 |

### 提醒内容

语音默认播报：

> 「提醒：明天上午十点客户拜访」

用户要求详情时继续播报地点和备注。

IM 回执中的完整内容：

```
📌 提醒：客户拜访
━━━━━━━━━━━━━━━━━
时间：明天（7/22）10:00 - 12:00
地点：A 座 1503
备注：带合同复印件、身份证
━━━━━━━━━━━━━━━━━
你的回复：[已关闭]
━━━━━━━━━━━━━━━━━
可回复：
「知道了」→ 关闭提醒
「X 分钟后提醒」→ 推迟
```

IM 回执中的回复只作用于当前消息对应的提醒，不支持开放式日历查询。

### P2 与 P3 的边界

```
用户收到提醒
       │
       ├── 用户说「知道了」「不用再响了」「X 分钟后提醒」
       │   └── P2：只改提醒状态，不改日程
       │
       ├── 用户说「改到今晚八点」「今天不参加了」「这周跳过」「停止后续」
       │   └── P3：修改日程内容或周期规则（高风险确认）
       │
       ├── 用户说「再详细说说」
       │   └── P2：继续播报完整内容
       │
       └── 用户说无法识别的命令
           └── 系统回复可用操作提示
```

### 异常处理摘要

| 异常场景 | 用户看到什么 |
|---|---|
| 语音失败，IM 成功 | IM 有提醒记录；用户通过 IM 看到本次提醒 |
| IM 失败，语音成功 | 语音正常播报；后续重试 IM |
| 用户未回应 | 不重播；IM 记录保留 |
| 多条提醒同时到达 | 语音依次播报；IM 分别留存 |
| 多条提醒播完后用户仅说「知道了」 | 系统列举各提醒标题，请用户指定——「请问你指的是「吃药」还是「出门开会」？」 |
| 同一提醒连续推迟 3 次 | 第 3 次推迟到期后，标记为「已多次推迟」，不再播报 |
| 推迟后再次触发 | 语音播报开头注明「这是之前推迟过的提醒」 |

### 用户使用路径

```text
P1 创建的日程到达提醒时间
       │
       ▼
语音 + IM 同时发送（独立通道）
       │
       ▼
用户正在忙
       │
       ├── 用户不回应 → IM 保留记录，不重播
       │
       ├── 用户说「知道了」「不用再响了」
       │   └── 提醒关闭 → IM 回执 → 日程不变
       │
       ├── 用户在 IM 中回复「知道了」或「X 分钟后提醒」
       │   └── 提醒关闭/推迟 → IM 回执更新（与语音效力等同）
       │
       ├── 用户说「X 分钟后提醒」
       │   └── 推迟，到时间后再次触发
       │       └── 最多 3 次 → 标记已多次推迟
       │
       └── 用户说「改时间」「跳过」
           └── 告知转入 P3
```

## 原型 / Demo

### 场景零：IM 回复关闭提醒

**现状**

小林收到推送提示「14:30 客户拜访」，但他正在安静的会议室，不方便说话。

**提议后的体验**

小林在 IM 回执中找到这条提醒，点击「知道了」按钮（或回复文字「知道了」）。系统更新提醒状态为已关闭，IM 回执更新为「你的回复：[已关闭]」。底层日程不变。

### 场景一：稍后提醒

**现状**

小林设置了「明天上午十点客户拜访」14:30 提醒。14:30 时他正在和同事开会，手机响了一声。

**提议后的体验**

14:30，小林手机语音播报：「提醒：明天上午十点客户拜访」。同时 IM 收到一条提醒消息。

小林不方便看屏幕，直接说了一句「十分钟后提醒我」。

系统追问时间时小林未指定时长，系统问「多久后再提醒？」小林回「十分钟」。系统回复（语音）：「好的，已推迟至 14:40」。

14:40，系统再次语音播报：「提醒：这是之前推迟过的提醒——明天上午十点客户拜访」。

**只发生了什么变化**

| 对象 | 变化 |
|---|---|
| 提醒状态 | 已推送 → 已推迟（稍后再次触发）→ 已推送 |
| 日程「客户拜访」 | 不变（时间仍然是明天上午十点） |

### 场景二：关闭提醒 + IM 回执

**现状**

小林早上收到「8:00 提醒吃药」。他正好刚吃完药。

**提议后的体验**

语音播报：「提醒：8:00 吃药」。小林说「知道了」。

系统回复（语音）：「好的，已关闭」。同时 IM 推送一条回执：

```
📌 提醒：吃药
━━━━━━━━━━━━━━━━━
时间：今天（7/22）8:00
```

**只发生了什么变化**

### 场景三：用户意图转入 P3

**现状**

小林有「每周五下午五点提交周报」的周期提醒。这周五他请假了，不需要交。

**提议后的体验**

周五 16:30 语音播报：「提醒：今天 17:00 提交周报」。小林说「这周不交了」。

系统回复（语音）：「这是跳过这次周期实例的操作，将为你转入日程修改流程」。引导进入 P3。

**只发生了什么变化**

| 对象 | 变化 |
|---|---|
| P2 | 不处理。系统识别为 P3 意图，引导跳转 |
| 提醒状态 | 不变（仍为已推送） |

### 场景四：并发提醒的歧义确认

**现状**

小林同时有「8:00 吃药」和「9:00 出门开会」两条提醒。8:00 时两条同时触发。

**提议后的体验**

语音依次播报：「提醒：8:00 吃药」「提醒：9:00 出门开会」。小林听后说「知道了」。系统回复：「请问你指的是「吃药」还是「出门开会」？」。小林说「吃药」。系统关闭「吃药」的提醒，「出门开会」保持已推送状态。

### 异常场景示例

- 多条提醒同时到达：语音依次播报「提醒：8:00 吃药」「提醒：9:00 出门开会」；IM 分别留存两条记录。
- 用户无回应且语音通道失败：IM 保留提醒记录，用户之后可翻看。
- 同一提醒第三次推迟到期：语音播报时注明「这是已多次推迟的提醒」，之后不再播报。

### 边界与非法情况

1. 用户说「晚点再说」→ 未指定时长，系统追问「多久后再提醒？」。
2. 用户说「先放着」→ 系统回复「无法识别，请说：知道了 / X 分钟后提醒」。不映射为关闭。
3. 用户说「改到今晚八点」→ 检测为 P3 意图，回复「这是修改日程操作，将为你转入日程修改流程」。
4. 用户说「这个月都不交了」→ 检测为 P3 意图（修改周期规则），引导进入 P3。
5. 用户说「今天不参加了」→ 检测为 P3 意图（跳过周期实例），引导进入 P3。
6. 语音播报时用户说话打断 → 停止播报，等待用户指令。用户指令属于 P2 则执行，属于 P3 则引导。
7. IM 消息发送失败 → 语音正常播报；系统后台重试 IM。
8. 用户对已关闭的提醒再次说「知道了」→ 系统回复「此提醒已关闭」。
9. 多条提醒播完后用户说「知道了」→ 系统列举提醒标题并询问目标。
10. 用户在 IM 回执中回复「知道了」→ 与语音「知道了」具有同等效力，只作用于该回执对应的提醒。

## 验收标准

1. P1 创建的日程到达提醒时间后，提醒同时通过语音和 IM 通道触发。一条通道失败不阻塞另一条。
2. 用户说「知道了」或「不用再响了」后，提醒状态变为已关闭；底层日程不变。
3. 用户说「十分钟后提醒」后，系统追问缺失时长或直接使用用户指定的时间，提醒推迟后再次触发；底层日程不变。
4. 用户说「这次不做了」或「今天不参加了」，系统不执行操作，告知将转入 P3 流程。
5. 语音默认播报标题和发生时间；用户要求详情时继续播报完整内容；地点和备注通过 IM 回执展示。
6. 用户未指定推迟时长时，系统追问「多久后再提醒？」。用户有预设默认值时除外。
7. 用户未回应时，系统不重复语音播报；IM 记录保留。
8. 多条提醒同时到达时，语音依次播报，IM 分别留存。
9. 同一提醒推迟最多 3 次，第 3 次后不再播报，标记为「已多次推迟」。
10. 推迟后的提醒再次触发时，语音注明「这是之前推迟过的提醒」。
11. 每次提醒触发和用户响应均生成 IM 回执，回执只绑定当前提醒，不提供日历查询指令。
12. 提醒状态变化不影响底层日程的任何字段。
13. Demo 能跑通：P1 创建日程 → 到达提醒时间 → 语音+IM 触发 → 用户一句话响应 → 确认提醒状态变更但日程不变。
14. Demo 能跑通异常场景：通道失败、多条同时到达、连续推迟超限。
15. 推迟后的提醒再次触发时，语音注明「这是之前推迟过的提醒」。
16. Demo 能展示：同一提醒推迟 3 次后不再播报。
17. 多条提醒同时到达、用户仅说「知道了」时，系统列举各提醒标题要求用户指定目标。
18. IM 回执中的「知道了」「X 分钟后提醒」回复与语音具有同等效力，只改变该回执对应提醒的状态。
19. 用户在 IM 中回复关闭提醒后，该提醒关闭，其他提醒不受影响。
20. Demo 能展示并发提醒的歧义确认流程：语音依次播报 → 用户说「知道了」→ 系统列举目标 → 用户指定 → 一条关闭、另一条保持已推送。
