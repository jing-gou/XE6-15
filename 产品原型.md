# 产品原型
> 基于：`Proposal 01-任务池导入`、`Proposal 02-优先级排序及展示`、`Proposal 03-任务自动重排`
> 时间：2026-07-14
> 目标：MVP 原型展示，核心逻辑梳理

## Proposal 依赖关系
三个 Proposal 的依赖关系：

```
GitHub Issue / PR / Review Request
        │
        ▼
  Proposal 01：任务池导入 ── 把散落在各处的个人行动收进一个池子
        │
        ▼
  Proposal 02：优先级排序与展示 ── 告诉我今天先做什么，并解释为什么
        │
        ▼
  Proposal 03：任务自动重排 ── 计划失效后，自动给出新的可执行计划
        │
        ▼
  用户确认，回到 GitHub 执行
```

## 数据模型

### 任务对象（从 GitHub 同步）

| 字段 | 类型 | 来源 | 说明 |
|------|------|------|------|
| `id` | string | 系统生成 | 任务唯一标识 |
| `github_type` | `issue` \| `pr` \| `review_request` | GitHub | 区分三种行动类型 |
| `github_number` | number | GitHub | Issue/PR 编号 |
| `github_url` | string | GitHub | 跳转回原页面处理 |
| `title` | string | GitHub | 标题 |
| `repository` | string | GitHub | 所属仓库 full_name |
| `milestone` | string \| null | GitHub | 关联 Milestone 标题 |
| `labels` | string[] | GitHub | 标签列表 |
| `github_status` | `open` \| `closed` \| `merged` | GitHub | GitHub 侧状态 |
| `created_at` | datetime | GitHub | 创建时间 |
| `updated_at` | datetime | GitHub | 最后更新时间 |

### 个人计划信息（用户侧，不写回 GitHub）

| 字段 | 类型 | 来源 | 说明 |
|------|------|------|------|
| `priority` | `P0` \| `P1` \| `P2` \| `unset` | 用户手动设置 | 个人优先级判断 |
| `hard_deadline` | date \| null | 用户设定 / Milestone 推算 | 硬截止日期 |
| `estimated_hours` | number \| null | 用户手动填写 | 预估投入时间 |
| `manual_order` | number \| null | 系统记录 | 用户拖拽后的手动位置 |
| `aging_days` | number | 系统计算 | 自导入或上次更新以来的天数（用于无 DDL 老化上浮） |

### 任务运行态

| 字段 | 类型 | 说明 |
|------|------|------|
| `work_status` | `on_progress` \| `pending_review` \| `done` | **主状态** — 决定是否参与可执行计划 |
| `status_reason` | `changes_requested` \| `blocked` \| `missing_estimate` \| `merged` \| `closed_unmerged` \| null | **状态原因标签** — 解释为什么处于当前状态/位置 |
| `plan_bucket` | `today` \| `planning_pool` \| `pending_review` \| `blocked` \| `unestimated` \| `done` | **计划位置** — 任务在界面中的归属区域 |

### 容量数据

| 字段 | 类型 | 说明 |
|------|------|------|
| `date` | date | 日期 |
| `total_hours` | number | 用户设定的当日可用总时长 |
| `used_hours` | number | 已完成任务的预计投入之和（不要求用户补录实际工时） |
| `remaining_hours` | number | `total_hours - used_hours` |

## work_status 主状态机

```mermaid
stateDiagram-v2
    direction LR

    [*] --> on_progress : 任务导入

    state on_progress {
        [*] --> 参与排序和重排
        参与排序和重排 --> 可进入今日计划
    }

    state pending_review {
        [*] --> 等待Reviewer
        等待Reviewer --> 离开可执行计划
    }

    state done {
        [*] --> 已合并_or_已关闭未合并
    }

    on_progress --> pending_review : 提交 PR / 完成返工并请求 Review
    pending_review --> on_progress : 收到 changes_requested\n或明确的未解决修改意见
    on_progress --> done : Issue 关闭 / PR 合并 / PR 关闭未合并
    done --> on_progress : Issue 或未合并 PR 被 Reopen\n（保留原有优先级/DDL/预估/手动顺序）

    note right of pending_review
        Approved 或普通评论
        不改变此状态
    end note

    note right of done
        PR 合并 → "已合并"
        PR 关闭未合并 → "已关闭未合并"
    end note
```

## 排序算法流程

```mermaid
flowchart TD
    Start(["用户点击排序 / 触发重排"]) --> Step1

    subgraph Step1["第一步：紧急置顶"]
        A1["筛选 DDL ≤ 24h 的任务"]
        A2["按 DDL 近 → 远排列"]
        A1 --> A2
    end

    Step1 --> Step2

    subgraph Step2["第二步：按优先级分组"]
        B1["P0 组（卡住或核心功能）"]
        B2["P1 组（重要但能商量）"]
        B3["P2 组（锦上添花）"]
        B4["未设置优先级 → 归入 P2 组末尾"]
        B1 --> B2 --> B3 --> B4
    end

    Step2 --> Step3

    subgraph Step3["第三步：组内按 DDL 排"]
        C1{"有 DDL？"}
        C1 -->|"是"| C2["按截止日期近 → 远"]
        C1 -->|"否"| C3["按老化因子上浮"]
        C3 --> C3a["&lt; 7 天：不动"]
        C3 --> C3b["7-14 天：轻微上浮"]
        C3 --> C3c["14-21 天：明显上浮"]
        C3 --> C3d["&gt; 21 天：大幅上浮"]
    end

    Step3 --> Step4

    subgraph Step4["第四步：同 DDL 内按预估用时排"]
        D1{"有预估用时？"}
        D1 -->|"是"| D2["用时短 → 长"]
        D1 -->|"否"| D3["排该档最后"]
    end

    Step4 --> Output

    subgraph Output["输出"]
        O1["排序后的任务列表"]
        O2["每项附带排序理由"]
    end

    Output --> End(["展示结果"])
```

## 自动重排流程

```mermaid
flowchart TD
    Trigger["状态变化发生<br/>（GitHub Webhook / 用户操作）"]

    Trigger --> Classify{"判断变化类型"}

    Classify -->|"新任务进入仓库"| NewTask["新任务根据既有优先级<br/>参与排序和重排"]
    Classify -->|"提交 PR / 完成返工"| ToReview["work_status → pending_review<br/>离开可执行计划<br/>进入待评审区域"]
    Classify -->|"收到 changes_requested"| ToProgress["work_status → on_progress<br/>回到可执行计划<br/>参与排序和重排"]
    Classify -->|"任务完成"| ToDone["work_status → done<br/>退出可执行任务池<br/>按预计投入扣减容量"]
    Classify -->|"任务阻塞"| ToBlocked["plan_bucket → blocked<br/>离开可执行计划<br/>进入阻塞区域"]
    Classify -->|"截止变化"| DeadlineChange["更新 hard_deadline<br/>重新参与排序"]
    Classify -->|"解除阻塞"| Unblock["plan_bucket → planning_pool<br/>重新参与排序竞争"]

    NewTask --> Reschedule
    ToProgress --> Reschedule
    DeadlineChange --> Reschedule
    Unblock --> Reschedule

    subgraph Reschedule["重新形成计划"]
        direction TB
        R1["按排序算法重新计算顺序"]
        R2["按容量分配任务到今日计划"]
        R3{"硬截止任务<br/>超过今日容量？"}
        R1 --> R2 --> R3
        R3 -->|"否"| Auto["计划自动更新<br/>用户直接查看变化"]
        R3 -->|"是"| Conflict["弹出容量冲突界面<br/>由用户选择处理方向"]
    end

    ToReview --> ShowPending["在待评审区域展示"]
    ToDone --> ShowDone["显示结束结果<br/>已合并 / 已关闭未合并"]
    ToBlocked --> ShowBlocked["在阻塞区域展示"]

    Auto --> Summary
    Conflict -->|"用户选择后"| Summary

    subgraph Summary["变化摘要"]
        S1["+ 哪些任务被加入"]
        S2["- 哪些任务被移出"]
        S3["↑↓ 哪些任务顺序变化"]
        S4["每项变化的原因"]
    end

    Summary --> Display["今日计划页展示<br/>新计划 + 变化摘要"]
```

## 关键决策

```mermaid
mindmap
  root((MVP 关键决策))
    数据边界
      GitHub-only 唯一数据源
      只读，不写回
      个人计划信息不写回 GitHub
    排序
      透明规则，逐条可核对
      优先级 → DDL → 预估用时
      无 DDL 老化上浮
      AI 仅作后置调整参考
    重排
      自动生效，不需逐项确认
      统一当日与跨日体验
      唯一例外：硬截止容量冲突
    主状态
      只有三个：On Progress / Pending Review / Done
      其他用 status_reason 标签
      Approved 不改变 Pending Review
    容量
      按预计投入扣减
      不要求用户填实际工时
      硬截止超出时由用户决定
    不做
      不做团队视图
      不做多数据源
      不做 AI 黑盒排序
      不做长期倒排/甘特图
```
