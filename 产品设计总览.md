# VoiceLife 声活 — 产品设计总览

## 产品定位

VoiceLife 声活是一款**语音优先、IM 辅助**的日程提醒工具。用户通过语音说出日程或小事，在约定的时间点收到语音提醒，随后可以关闭、推迟或修改已有安排——全程以语音交互为主，IM 提供可回看的操作回执。

产品围绕一条连续体验展开：**创建 → 提醒触发 → 即时响应 → 变更与周期控制**。

## 目标用户

- 有日程管理需求、希望通过语音而非手动操作完成交互的用户
- 通勤、走路、做饭等不便手动操作时刻的用户
- 无屏幕或屏幕受限设备的使用者

## 核心问题与价值

脑子里记不住小事、手动操作太繁琐、现有工具交互成本高。VoiceLife 把「记、提醒、改」三个动作收敛为语音对话——用户随口说、App 按时提醒、用户随口回应即可完成闭环。

## MVP 范围

MVP 覆盖三个 Proposal，对应产品体验链路上的三个关键环节：

| Proposal | 负责的环节 | 核心能力 |
| --- | --- | --- |
| Proposal 1：语音创建与查询 | 输入端 | 通过语音创建日程/提醒事项、查询已有事项 |
| Proposal02：提醒触发与即时响应 | 提醒端 | 在约定时间触发语音+铃声提醒，用户可通过语音打断、推迟、追问详情 |
| Proposal03：已有日程变更与周期控制 | 变更端 | 语音修改、跳过、删除已有日程及周期事项，支持 15 分钟内撤销 |

三者串起一条完整链路：用户说出一个日程 → 时间到了被提醒 → 随口回应处理 → 必要时修改或撤销。

## MVP 明确不做

- 位置、蓝牙、Wi-Fi 等非时间的触发条件
- 特殊方言及多语言语音响应识别
- 用户自定义唤醒词
- 复杂的周期规则（工作日/节假日/调休日等）
- 跨日历源同步（Google/Exchange/飞书等）
- 跨时区与夏令时处理
- 批量操作及第三方平台数据接入

## 关键跨 Proposal 决策

| 决策项 | 选择 | 简要理由 |
| --- | --- | --- |
| 交互方式 | 语音为主，IM 为操作回执 | 语音负责实时快速交互，IM 负责提供完整可回看的记录 |
| 设备假设 | App 始终前台运行 | 暂无后台/锁屏唤醒需求，后续可扩展 |
| 唤醒词 | 固定唤醒词"小鹏小鹏" | 降低误识别率，自定义唤醒词后续 Proposal 覆盖 |
| 触发条件 | 仅时间触发 | 时间为日程基本必要字段，非时间触发条件后续扩展 |
| 平台策略 | 先做无 GUI 场景（纯语音），GUI 为辅 | 语音主路径走通后再补 GUI 增强 |

## 基本概念（跨 Proposal 共用）

| 概念 | 定义 | 负责 Proposal |
| --- | --- | --- |
| 日程 | 有**时间段**的事项（如"下午三点到五点开会"） | Proposal 1 |
| 提醒事项 | 仅有**时间点**的事项（如"七点吃药"） | Proposal 1 |
| 预告 | 开始前 15min 的短铃声+语音提醒 | Proposal02 |
| 提醒 | 开始时的语音+持续铃声提醒，1min 无回应则推迟 | Proposal02 |
| 推迟提醒 | 提醒触发后约定再提醒，不改变原定时间 | Proposal02 |
| 单次日程 | 只发生一次的日程 | Proposal03 |
| 周期事项 | 按固定规则重复的事项 | Proposal03 |
| 撤销 | 15 分钟内可恢复最近一次变更 | Proposal03 |
